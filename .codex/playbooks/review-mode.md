# Review Mode

## 目标

当用户要求“review”时，优先识别风险、回归、缺口和治理问题，而不是先写总体评价。

review 的重点不是“这份代码看起来不错”，而是：

1. 有没有行为风险
2. 有没有契约与语义漂移
3. 有没有测试缺口
4. 有没有重复出现的低质量模式

## 输出顺序

1. Findings
2. Open questions / assumptions
3. 简短总结

禁止先写大段总体评价。

## Findings 要求

- 按严重度排序
- 尽量指出文件路径与具体位置
- 优先写明确可行动的问题，而不是抽象感受
- 如果问题会导致回归、误导用户、破坏契约或留下治理漏洞，就应当作为 finding，而不是“建议”

若 review 对象属于页面、模块、函数或接口，先按 `docs/governance/AGENT_OUTPUT_PROTOCOL.md` 补齐关键代码位置、真实流程、`逻辑矫正` 与注释问题，再抽 findings；不要只看表面 diff。

## 优先关注的风险面

优先关注：

- 行为回归
- 数据一致性
- 多租户隔离
- 权限
- 支付 / 资金 / 认证风险
- 契约与类型漂移
- 用户可见语义漂移
- 批量语义与删除语义
- 缺失测试或低质量测试

说明：

- 通用反模式分类优先参考根 `AGENTS.md` §5（证据等级与假绿禁止）+ §0 硬规则。
- review-mode 只强调“这些问题在 review 时如何升级为 finding”，不重复维护仓库总反模式清单。

## 明确要抓的 review 升级信号

遇到以下模式，应优先判断是否构成 finding：

### 1. 契约与类型类

- 命中契约/类型反模式（前端手写 backend 同义 DTO/VO、跳过 `pnpm generate-types`、未走 OpenAPI 链路）
- backend 契约改动后未评估 `generate-types` 与前端传播面

### 2. 用户可见语义类

- 命中用户可见语义反模式（原始 code/英文枚举直出、多展示面文案漂移、`xxxText` 未确认）
- 直接渲染 `statusText`、`teamStatusText`、`xxxText` 却没有契约保证

### 3. 批量与事务类

- 明明可批量处理却写成逐条调用
- 把事务中的 `Promise.all` 当成并行优化理由

### 4. 测试质量类

- 只有 smoke，没有行为测试
- 改了 `apps/admin-web/src/views/**` 却没有 `pnpm verify:admin-view-types` 证据
- 高风险链路只有 happy path，没有边界或失败路径
- 测试文件大量依赖 `@ts-nocheck` 或 `any`

### 5. 高风险治理类

- 多租户、权限、支付、认证、字典、任务、Prisma 改动缺少专项检查
- 跨 app 契约改动只修一端
- 高风险改动夹带无关重构

## Open questions / assumptions

只有在以下情况才放到这一节，而不是直接写 finding：

- 本地证据不足以证明问题一定存在
- 行为依赖未展示的配置、上游返回或运行时环境
- 某个实现可能是刻意设计，但当前上下文无法确认

这部分应简短、具体，不要把明显问题降级成“开放问题”。

## 无发现时

若未发现明确问题，必须明确写：

- “未发现明确问题”

同时仍需补一句：

- 残余风险
- 未验证项
- 上下文不足导致的盲区

## 不应做的事

- 不先写大段总体评价
- 不把“没跑验证”说成“没问题”
- 不用泛泛而谈代替具体风险点
- 不把明显缺陷包装成“可以延期处理”
- 不因为代码量大就忽略重复错误模式
