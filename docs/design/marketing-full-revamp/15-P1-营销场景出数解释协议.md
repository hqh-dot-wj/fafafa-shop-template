# 1. 文档目的

本文承接：

- [ADR-0005 营销 C 端以场景发布快照为主链路](../../adr/ADR-0005-marketing-scene-driven-resolution.md)
- [14-C-compatible-B-营销架构演进方案](./14-C-compatible-B-营销架构演进方案.md)

P1 阶段不做完整推荐系统，也不引入重型活动中台。P1 要先解决一个更基础的问题：

```text
同一个首页、不同账户、不同手机、不同渠道看到的商品为什么不同？
```

这个问题不能只用“推荐算法不同”解释。当前项目更多是场景发布快照 + 模块裁决 + 商品权益解析 + 小程序缓存 / 降级共同作用。因此 P1 的目标是建立一套可审计、可复现、可排障的“场景出数解释协议”。

# 2. 核心结论

场景出数差异应先被分成三类：

| 类型     | 示例                                                                       | 结论                                                       |
| -------- | -------------------------------------------------------------------------- | ---------------------------------------------------------- |
| 合理差异 | 不同账户的新客状态、会员等级、可用优惠券、地域、受众策略不同               | 允许，但必须能解释                                         |
| 边界差异 | 同一账户在不同手机、不同渠道、不同版本看到个别商品不同                     | 允许存在，但只能由渠道、版本、缓存、降级、发布时间窗口解释 |
| 异常差异 | 同一账户、同一渠道、同一版本、同一发布号，在短时间内看到无法解释的不同商品 | 不应接受，应进入排障                                       |

所以 P1 的设计重点不是“让每个人看到完全一样”，而是：

```text
让每一次不同都有原因、证据和归因边界。
```

# 3. 当前事实

## 3.1 已有能力

当前代码已经具备 P1 的基础：

- C 端主入口：`apps/backend/src/module/client/marketing/scene/client-scene.controller.ts`
- 场景裁决服务：`apps/backend/src/module/marketing/resolution/resolution.service.ts`
- 场景发布快照：`MktSceneRelease.releaseSnapshot`
- 场景模块定义：`MktSceneModule`
- 裁决审计表：`MktResolutionAudit`
- 观测指标：`ResolutionObservabilityService`
- 小程序场景请求：`apps/miniapp-client/src/api/marketing.ts`
- 小程序缓存 / fallback：`apps/miniapp-client/src/composables/use-scene-marketing.ts`

现在 `resolveSceneView` 已经记录：

- `traceId`
- `sceneCode`
- `releaseNo`
- `channel`
- `durationMs`
- `moduleCount`
- `emptyModuleCount`
- 成功 / 失败状态

也已经有模块级裁决链路：

```text
release snapshot
  -> candidate load
  -> audience filter
  -> primary offer resolve
  -> rank
  -> card build
```

## 3.2 主要缺口

当前缺口不在于没有链路，而在于链路中的差异原因没有被结构化记录：

- 不知道某个商品为什么进入候选集。
- 不知道某个商品为什么被过滤。
- 不知道排序前后为什么变化。
- 不知道用户上下文里哪些字段参与了裁决。
- 不知道小程序最终展示的是实时场景、旧缓存、兼容聚合 fallback，还是空结果。
- 不知道“同一账户不同手机”到底是渠道差异、版本差异、缓存差异，还是服务端异常。

# 4. 目标模型

P1 建议引入四个概念：

```text
SceneResolveContext
SceneResolveFingerprint
SceneResolveTrace
SceneResolveExplain
```

## 4.1 SceneResolveContext

`SceneResolveContext` 是一次场景出数请求的上下文快照。它回答：

```text
这次请求是谁、在哪里、用什么客户端、按哪个发布号、在什么时间请求的？
```

建议字段：

| 字段          | 来源               | 说明                                 |
| ------------- | ------------------ | ------------------------------------ |
| tenantId      | 后端租户上下文     | 多租户隔离基础                       |
| memberId      | C 端登录态         | 用户身份                             |
| sceneCode     | path param         | 场景编码                             |
| channel       | query / 默认值     | `MINIAPP / H5 / ADMIN_PREVIEW`       |
| clientVersion | header 或 query    | 小程序 / H5 版本                     |
| platform      | header 或 query    | iOS / Android / WeChat / H5          |
| appBuild      | header 或 query    | 更细的构建号，可选                   |
| deviceHash    | header 或 query    | 设备弱标识，只用于排障，不作为强身份 |
| requestTime   | 服务端时间         | 不信任客户端时间                     |
| releaseNo     | 发布快照           | 实际命中的发布号                     |
| moduleLimit   | query              | 模块截断参数                         |
| productLimit  | query              | 商品截断参数                         |
| traceId       | 后端生成或透传     | 排障主键                             |
| cacheSource   | 前端回传或后端返回 | scene / fallback / stale-cache       |

注意：`deviceHash` 只能做排障辅助，不应用它决定强权益。优惠券、积分、购买资格这类权益必须跟账户或订单绑定。

## 4.2 SceneResolveFingerprint

`SceneResolveFingerprint` 是上下文指纹。它回答：

```text
这两次请求是否应该被认为是同一类裁决？
```

建议由以下字段生成：

```text
tenantId
memberId
sceneCode
channel
clientVersion
platform
releaseNo
moduleLimit
productLimit
audienceContextVersion
entitlementContextVersion
```

需要刻意排除：

- `traceId`：每次请求都不同，不能参与稳定性判断。
- `requestTime`：可参与解释，但不宜直接进入主指纹，否则无法判断稳定性。
- `deviceHash`：默认不进入主指纹，除非未来明确设计设备级投放。

关键判断：

```text
相同 fingerprint + 相同 releaseNo + 无库存/容量变化
=> 短时间内应得到一致结果。
```

如果不一致，要么是缓存 / fallback，要么是排序用了非确定性因素，要么是裁决链路有隐藏状态。

## 4.3 SceneResolveTrace

`SceneResolveTrace` 是机器排障用的结构化轨迹。它不直接面向 C 端展示，主要给后台、日志和测试使用。

建议分层：

```text
request
release
modules[]
  candidates[]
  filters[]
  offers[]
  ranking[]
  selected[]
fallback
metrics
```

模块级 trace 示例：

```json
{
  "moduleCode": "HOME_FEATURED",
  "sourcePolicyCode": "HOME_PRODUCT_POOL",
  "audiencePolicyCode": "NEWCOMER_ONLY",
  "resolverPolicyCode": "PRIMARY_OFFER",
  "sortPolicyCode": "WEIGHT_DESC",
  "candidateCount": 20,
  "filteredCount": 8,
  "selectedCount": 6,
  "filterReasons": [
    { "code": "AUDIENCE_NOT_MATCH", "count": 5 },
    { "code": "COUPON_NOT_ELIGIBLE", "count": 2 },
    { "code": "STOCK_UNAVAILABLE", "count": 1 }
  ]
}
```

## 4.4 SceneResolveExplain

`SceneResolveExplain` 是给人看的解释结果。它不要求暴露所有底层字段，而是给出可理解结论：

```text
该用户命中新客首页发布号 12。
模块 HOME_FEATURED 从 20 个候选商品中选择 6 个。
8 个商品被过滤，其中 5 个因不符合新客策略，2 个因权益不可用，1 个因库存不可售。
本次结果来自 scene 实时裁决，没有使用 fallback。
```

后台可以看完整解释；C 端默认只返回 `traceId / releaseNo / source`，不展示敏感规则。

# 5. 差异归因协议

## 5.1 原因分类

P1 推荐固定一组原因码，先不要让运营自定义原因码。

| 分类     | 原因码                     | 说明                     |
| -------- | -------------------------- | ------------------------ |
| 发布     | `RELEASE_NOT_FOUND`        | 场景未发布               |
| 发布     | `RELEASE_CHANNEL_MISMATCH` | 发布快照不匹配渠道       |
| 上下文   | `CONTEXT_MEMBER_LEVEL`     | 会员等级导致差异         |
| 上下文   | `CONTEXT_NEWCOMER`         | 新客状态导致差异         |
| 上下文   | `CONTEXT_CLIENT_VERSION`   | 客户端版本导致差异       |
| 受众     | `AUDIENCE_NOT_MATCH`       | 受众策略不匹配           |
| 权益     | `COUPON_NOT_ELIGIBLE`      | 优惠券不可领 / 不可用    |
| 权益     | `POINTS_NOT_ENOUGH`        | 积分条件不满足           |
| 库存     | `STOCK_UNAVAILABLE`        | 实物库存不可售           |
| 容量     | `CAPACITY_UNAVAILABLE`     | 服务容量不可预约         |
| 排课     | `SCHEDULE_UNAVAILABLE`     | 无可履约课次             |
| 虚拟补位 | `VIRTUAL_FILL_POLICY`      | 受虚拟补位策略影响       |
| 排序     | `RANK_WEIGHT`              | 权重排序                 |
| 排序     | `RANK_RECENCY`             | 新近排序                 |
| 排序     | `RANK_PERSONALIZED`        | 个性化排序               |
| 限制     | `LIMIT_MODULE`             | 模块数被截断             |
| 限制     | `LIMIT_PRODUCT`            | 商品数被截断             |
| 缓存     | `CACHE_HIT`                | 使用缓存                 |
| 缓存     | `CACHE_STALE`              | 使用旧缓存               |
| 降级     | `FALLBACK_AGGREGATE`       | 使用兼容聚合 fallback    |
| 降级     | `MODULE_TIMEOUT`           | 模块裁决超时，返回空模块 |
| 灰度     | `ROLLOUT_BLOCKED`          | 放量比例未命中           |
| 熔断     | `SCENE_DISABLED`           | 场景开关关闭             |

原因码要稳定，展示文案可以在后台再映射。

## 5.2 同一账户不同手机

这是 P1 最需要深挖的问题。

同一账户、不同手机出现个别商品差异，合理解释只应来自：

- `channel` 不同：例如一个是 `MINIAPP`，一个是 `H5`。
- `platform` 不同：例如 iOS / Android 对某些卡片、跳转或售卖能力不同。
- `clientVersion` 不同：旧版本不支持某些模块或卡片模板。
- `cacheSource` 不同：一台手机展示实时 scene，另一台手机展示旧缓存。
- `fallback` 不同：一台命中场景接口，另一台因为超时走兼容入口或本地缓存。
- `requestTime` 跨过发布窗口：一台在发布前请求，一台在发布后请求。

不应接受的解释：

- “可能推荐不同”：当前链路还没有成熟个性化推荐引擎，不能用这个兜底。
- “设备不同所以不同”：除非明确设计设备级策略，否则设备只能解释缓存和版本，不应解释权益资格。
- “偶发”：如果同 fingerprint 下重复出现，必须能定位。

判断规则：

```text
同 memberId
+ 同 tenantId
+ 同 sceneCode
+ 同 channel
+ 同 clientVersion
+ 同 platform
+ 同 releaseNo
+ 短时间窗口
+ 无库存/容量变化
= 应展示一致商品集合和排序
```

如果商品有差异，后台解释页必须至少回答：

- 哪个模块不同。
- 哪些商品只在 A 出现。
- 哪些商品只在 B 出现。
- 差异发生在候选、过滤、权益解析、排序、截断、缓存还是降级阶段。

## 5.3 不同账户同一手机

不同账户看到不同商品是正常的，但差异仍需归因：

- 新客 / 老客不同。
- 会员等级不同。
- 优惠券领取状态不同。
- 积分余额或积分资格不同。
- 已购 / 未购状态不同。
- 门店、课程、服务可预约状态不同。
- 拼课队伍参与状态不同。

这里有一个重要取舍：

```text
账户级权益差异应该由权益资产解释，不应该由前端缓存解释。
```

因此小程序缓存 key 不能只按 `sceneCode`。至少应纳入：

- `tenantId`
- `memberId`
- `sceneCode`
- `channel`
- `releaseNo` 或场景数据版本

如果缓存无法拿到 `releaseNo`，至少要在响应落地时保存 `releaseNo`，下一次请求后对比版本。

## 5.4 不同渠道

不同渠道允许不同：

- `MINIAPP` 可以展示小程序卡片。
- `H5` 可以展示 H5 可跳转卡片。
- `ADMIN_PREVIEW` 不受灰度放量限制，应展示预览结果。

但渠道差异必须来自发布快照、模块配置或卡片能力，而不是前端私自过滤。

## 5.5 库存、服务容量、排课资源的差异

实物库存和服务容量不应混为一类。

| 类型       | 是否强一致                 | 影响场景出数                   | 解释方式                                    |
| ---------- | -------------------------- | ------------------------------ | ------------------------------------------- |
| 实物库存   | 更偏强一致                 | 售罄时不应继续强推             | `STOCK_UNAVAILABLE`                         |
| 优惠券库存 | 领取库存强约束，展示可弱化 | 可展示但领取失败需解释         | `COUPON_NOT_ELIGIBLE` / `STOCK_UNAVAILABLE` |
| 积分预算   | 不是库存，是预算和账务约束 | 预算不足时活动应降级或停止发放 | `POINTS_BUDGET_EXHAUSTED`                   |
| 服务容量   | 依赖预约/履约窗口          | 某时段不可约，不等于商品不可售 | `CAPACITY_UNAVAILABLE`                      |
| 排课资源   | 依赖老师、教室、时段和课次 | 无课次时不能承诺履约           | `SCHEDULE_UNAVAILABLE`                      |
| 虚拟补位   | 是玩法策略，不是真实库存   | 可影响拼课展示热度和成团状态   | `VIRTUAL_FILL_POLICY`                       |

这会影响首页推荐：

- 实物商品：重点看是否可售、是否售罄。
- 服务商品：重点看是否可预约、是否有服务容量。
- 课程商品：重点看是否有可履约课次。
- 拼课商品：还要看队伍状态、虚拟补位、成团概率和履约排课。

所以“商品展示不同”不一定是推荐问题，可能是资源类型不同导致的履约能力差异。

# 6. 后端设计建议

## 6.1 不急着新增大表

P1 不建议一上来新增很多表。当前已有 `MktResolutionAudit`，可先做两步：

1. 扩展审计语义：让审计快照可承载场景级 trace，而不只服务于结算预览、下单、模拟器。
2. 若数据量变大，再拆 `mkt_scene_resolve_trace`。

推荐演进：

```text
阶段 1：复用 mkt_resolution_audit 的 snapshot 结构，记录抽样 trace
阶段 2：新增 mkt_scene_resolve_trace，按 traceId 查询完整解释
阶段 3：冷热分离，只保留聚合指标和近期 trace
```

## 6.2 如果新增表，建议结构

建议表名：

```text
mkt_scene_resolve_trace
```

建议字段：

| 字段                    | 说明                            |
| ----------------------- | ------------------------------- |
| id                      | 主键                            |
| tenant_id               | 租户                            |
| trace_id                | 排障主键                        |
| member_id               | 会员                            |
| scene_code              | 场景                            |
| channel                 | 渠道                            |
| client_version          | 客户端版本                      |
| platform                | 平台                            |
| release_no              | 发布号                          |
| fingerprint             | 上下文指纹                      |
| request_snapshot        | 请求上下文 JSON                 |
| release_snapshot_digest | 发布快照摘要，不复制完整快照    |
| explain_snapshot        | 候选、过滤、排序、降级原因 JSON |
| result_digest           | 最终模块和商品 ID 摘要          |
| source                  | scene / fallback / cache        |
| status                  | success / failed / degraded     |
| duration_ms             | 耗时                            |
| create_time             | 创建时间                        |

索引建议：

- `(tenant_id, trace_id)`
- `(tenant_id, scene_code, create_time)`
- `(tenant_id, member_id, create_time)`
- `(tenant_id, fingerprint, create_time)`

保留策略：

- 本地开发：30 天。
- 线上默认：7 到 30 天可配置。
- 长期只保留聚合指标，不长期保存用户级 trace。

## 6.3 resolveSceneView 改造点

建议在 `resolveSceneView` 外围创建 trace collector：

```text
beginTrace(context)
  load release
  record release
  for each module:
    record candidates
    record filters
    record offers
    record ranking
    record selected
  record fallback/degrade
  endTrace(result)
```

关键点：

- trace collector 不应影响主链路成功。
- trace 写入失败只记录日志，不阻断场景接口。
- trace 是否完整采样可配置。
- 管理端模拟器必须强制完整 trace。
- C 端真实请求默认抽样，异常和降级请求强制记录。

## 6.4 模块级解释接口

建议后台提供：

```text
GET /admin/marketing/scene/resolve-traces/:traceId
POST /admin/marketing/scene/simulate
POST /admin/marketing/scene/compare
```

用途：

- `resolve-traces/:traceId`：查一次真实请求解释。
- `simulate`：运营输入用户、渠道、场景，模拟结果。
- `compare`：对比两个账户或两个设备上下文差异。

`compare` 是核心，因为用户真正问的通常不是“为什么 A 看到这个”，而是：

```text
为什么 A 和 B 看到的不一样？
```

## 6.5 隐私与安全

解释不能把所有规则暴露给 C 端：

- C 端响应可返回 `traceId / releaseNo / source`。
- 后台解释可展示规则命中摘要。
- 风控、黑名单、作弊、内部权重细节不应给 C 端。
- trace 中的手机号、openid、设备原始 ID 不落明文。

# 7. 小程序设计建议

## 7.1 响应字段

`ClientSceneView` 建议未来扩展为：

```ts
interface ClientSceneView {
  sceneCode: string;
  releaseNo: number;
  traceId?: string;
  source?: 'scene' | 'fallback' | 'cache' | 'stale-cache';
  modules: ClientModuleView[];
}
```

这不是为了给用户看，而是为了排障：

- 客服或运营可拿 `traceId` 查后台解释。
- 前端日志可知道本次展示来源。
- 出现首页差异时能判断是否缓存导致。

## 7.2 缓存 key

当前 `useSceneMarketing` 支持外部传入 `cacheKey`，这是灵活的，但也容易让调用方漏掉关键维度。

建议封装统一 cache key builder：

```text
sceneMarketing:${tenantId}:${memberId}:${sceneCode}:${channel}:${releaseNo}
```

如果请求前没有 `releaseNo`：

```text
sceneMarketing:${tenantId}:${memberId}:${sceneCode}:${channel}:latest
```

并在响应后保存：

```text
sceneMarketingMeta:${tenantId}:${memberId}:${sceneCode}:${channel}
```

meta 中保存：

- `releaseNo`
- `traceId`
- `source`
- `cachedAt`

## 7.3 前端不做隐式二次过滤

小程序可以做：

- 卡片模板兼容。
- 空图兜底。
- 跳转能力兜底。
- 接口失败 fallback。

小程序不应做：

- 按会员等级二次过滤营销商品。
- 按优惠券资格二次过滤。
- 按库存状态二次决策商品是否展示。
- 把不同接口来源的数据混排后不标来源。

否则后台无法解释“最终看到的结果”。

# 8. 后台设计建议

P1 后台不必先做复杂可视化，但至少要有三类能力。

## 8.1 Trace 查询

输入：

- `traceId`

输出：

- 请求上下文。
- 发布号。
- 模块列表。
- 每个模块候选数、过滤数、选中数。
- 主要过滤原因。
- 是否超时、fallback、缓存。

## 8.2 模拟器

输入：

- `memberId`
- `sceneCode`
- `channel`
- `clientVersion`
- `platform`
- `requestTime`

输出：

- 模拟展示结果。
- explain snapshot。

模拟器要明确标注：

```text
模拟结果不等于历史真实结果。
```

因为历史真实结果可能受到当时库存、缓存、发布号和临时故障影响。

## 8.3 差异对比

输入：

- trace A
- trace B

输出：

- 上下文差异。
- 发布号差异。
- 模块差异。
- 商品集合差异。
- 原因归因。

对比页最适合处理：

- 同一账户不同手机。
- 同一手机不同账户。
- 同一用户昨天和今天首页不同。
- 运营预览和 C 端真实展示不同。

# 10. 实施分析

P1 不是“补一个 traceId 字段”。P1 要把场景出数链路改造成可解释链路，但实施深度仍按阶段推进。

后端主链路：

- `apps/backend/src/module/client/marketing/scene/client-scene.controller.ts`
- `apps/backend/src/module/client/marketing/scene/client-scene.service.ts`
- `apps/backend/src/module/marketing/resolution/resolution.service.ts`
- `apps/backend/src/module/marketing/resolution/scene-release.repository.ts`
- `apps/backend/src/module/marketing/resolution/resolution-observability.service.ts`

- `apps/miniapp-client/src/composables/use-scene-marketing.ts`

现有链路已经有 `traceId`、`releaseNo`、场景发布快照、模块裁决、空模块统计和 fallback，但缺少结构化原因。

## 10.2 实施目标

P1 实施目标应同时覆盖三件事：

```text
请求上下文可复现
裁决过程可解释
前端展示来源可追踪
```

对应产物：

- `SceneResolveContext`：记录会员、租户、场景、渠道、版本、平台、发布时间。
- `SceneResolveFingerprint`：判断两次请求是否应该得到一致结果。
- `SceneResolveTrace`：记录候选、过滤、权益解析、排序、截断、降级。
- `SceneResolveExplain`：给后台和排障使用的人读解释。
- `ClientSceneView.traceId/source`：让 C 端展示结果能反查。

## 10.3 后端改造分析

### 入口层

`ClientSceneController.getSceneModules` 需要补齐上下文维度：

- `channel`
- `clientVersion`
- `platform`
- `appBuild`
- `deviceHash` 的脱敏版本
- `moduleLimit`
- `productLimit`

这些字段不一定全部参与裁决，但要进入 trace，方便解释“同一账号不同手机”的差异。

### 裁决层

`ResolutionService.resolveSceneView` 应创建 trace collector：

```text
beginTrace
  record context
  record release
  record module limit
  resolve each module
  record success / degrade / fail
endTrace
```

`resolveSingleModuleView` 需要把每一步原因显式抛给 collector：

- candidate loader：候选来源和数量。
- audience filter：受众过滤原因。
- primary offer resolver：优惠、库存、权益不可用原因。
- ranker：排序规则和截断原因。
- card builder：最终选中商品。

要求：

- collector 写入失败不阻断主链路。
- 异常、降级、空模块强制记录。
- 正常请求可抽样，但后台模拟器必须完整记录。

### 观测层

`ResolutionObservabilityService` 继续承担指标统计，不替代 trace。指标回答“是否异常”，trace 回答“为什么这样”。

两者关系：

```text
observability = 聚合指标
trace = 单次请求解释
```

## 10.4 小程序改造分析

`ClientSceneView` 建议扩展：

```ts
interface ClientSceneView {
  sceneCode: string;
  releaseNo: number;
  traceId?: string;
  source?: 'scene' | 'fallback' | 'cache' | 'stale-cache';
  modules: ClientModuleView[];
}
```

`useSceneMarketing` 需要统一展示来源：

- scene：实时场景接口。
- fallback：兼容接口或 fallbackLoader。
- cache：当前本地缓存。
- stale-cache：接口失败后使用旧缓存。

缓存 key 不能只看 `sceneCode`。建议形成统一 builder：

```text
tenantId + memberId + sceneCode + channel + releaseNo/dataVersion
```

如果请求前没有 `releaseNo`，要保存 meta：

- 上次 releaseNo。
- 上次 traceId。
- 上次 source。
- cachedAt。

## 10.5 契约影响

如果 C 端响应新增 `traceId / source`：

- backend DTO / VO 变化属于跨 app 契约变更。
- 需要 `pnpm generate-types`。
- miniapp-client 使用生成类型或同步共享类型，不手写同义契约绕过。

如果只先在日志 / Redis 内部记录 trace，不改响应：

- 可以先 backend-only。
- 但无法解决客服或用户现场反馈时反查 trace 的问题。

推荐：

```text
P1 应把 traceId 返回给 C 端。
source 可同时返回，方便判断缓存和 fallback。
```

## 10.6 数据影响

阶段化选择：

| 阶段 | 数据方式                       | 说明                         |
| ---- | ------------------------------ | ---------------------------- |
| P1-A | 日志 + Redis + 现有审计 JSON   | 先验证 trace 结构            |
| P1-B | 独立 `mkt_scene_resolve_trace` | 后台解释页需要稳定查询时引入 |
| P1-C | trace 聚合 + 采样策略          | 流量上来后控制数据量         |

P1-A 不是临时补丁，它要按目标 trace schema 写 collector，只是存储介质先不独立建表。

## 10.7 后台影响

后台建议分三步：

1. traceId 查询：按一次真实请求查看解释。
2. 模拟器：输入 memberId、sceneCode、channel、clientVersion 得到模拟解释。
3. 差异对比：trace A 对 trace B，解释两个用户或两个设备为什么不同。

不要一开始做运营可编辑推荐规则。P1 的后台是解释和诊断，不是推荐平台。

## 10.8 验证分析

必须验证：

- 同 fingerprint、同 releaseNo、无库存容量变化时结果稳定。
- 同一账号不同手机时，差异只能来自渠道、版本、平台、缓存、fallback、发布时间窗口。
- 不同账号差异能归因到受众、权益、库存、容量、拼课状态。
- 模块超时返回空模块时 trace 有 `MODULE_TIMEOUT`。
- fallbackLoader 成功时 source 是 fallback。
- stale cache 命中时 source 是 stale-cache。
- trace 写入失败不影响接口成功。

P1 不做：

- 不做推荐算法。
- 不做个性化模型。
- 不做 AB 实验平台。
- 不重构优惠券和积分账务。

# 12. 与推荐系统的边界

这里必须提出质疑：

```text
如果首页不同账户看到不同商品，是否就意味着我们需要推荐系统？
```

答案：不一定。

当前阶段更可能是以下因素造成：

- 场景模块策略不同。
- 受众策略不同。
- 权益资格不同。
- 库存 / 服务容量不同。
- 拼课队伍状态不同。
- 小程序缓存和 fallback 不同。

真正的推荐系统至少需要：

- 用户行为画像。
- 商品特征。
- 召回、粗排、精排。
- 实验分桶。
- 效果指标。
- 反作弊和冷启动策略。

这些不是 P1 的目标。P1 只为未来推荐系统预留解释接口：

```text
如果未来 rankingReason = RANK_PERSONALIZED
那么必须能记录模型版本、实验桶、召回源和排序摘要。
```

但在当前本地开发阶段，不建议把首页差异直接归因到“推荐算法”。

# 13. 风险与取舍

## 12.1 性能风险

trace 不能拖慢 C 端首页。

策略：

- 默认抽样。
- 异常强制记录。
- 管理端模拟强制完整记录。
- trace 写入异步化。

## 12.2 数据量风险

首页访问频繁，完整 trace 容易膨胀。

策略：

- 只保存摘要，不保存完整商品详情。
- 保存商品 ID、配置 ID、原因码、数量。
- 大 JSON 设置长度上限。
- 近期保留，过期删除。

## 12.3 隐私风险

trace 可能包含用户身份、权益状态、风控结论。

策略：

- 原始设备 ID 不落库。
- 手机号、openid 不进 trace。
- C 端不返回内部规则。
- 后台查询加权限。

## 12.4 解释漂移

代码逻辑变了，但解释原因没有同步更新，会导致解释不可信。

策略：

- 原因码由裁决代码产出，不由外层猜测。
- 新增过滤条件时必须新增或复用原因码。
- 单测断言原因码。

# 14. 逻辑矫正

需要先纠正几个容易误判的点：

1. “首页不同就是推荐系统”：不准确。当前更像场景策略、权益资格、库存容量、缓存降级共同导致的差异。
2. “不同手机看到不同就是正常”：不准确。同一账户不同手机只能由渠道、版本、平台、缓存、fallback、发布时间窗口解释，不能泛化为设备差异。
3. “traceId 已经有了，所以可解释”：不准确。`traceId` 只是索引，没有结构化原因就无法解释。
4. “前端缓存只是体验优化”：不准确。缓存会直接影响用户看到什么，必须纳入解释协议。
5. “服务商品和实物商品一样看库存”：不准确。服务看容量和履约窗口，课程还要看排课资源。

# 15. 注释审查与注释方案

如果后续进入代码实现，注释应集中在以下位置：

- `resolveSceneView`：说明 trace collector 不影响主链路，写入失败不阻断响应。
- `resolveSingleModuleView`：说明每个阶段要产出原因码，不允许外层猜测。
- cache key builder：说明 member、channel、releaseNo 等维度不能省略。
- trace DTO：说明哪些字段可返回 C 端，哪些字段仅后台可见。

不建议增加的注释：

- 不在每个原因码旁写大段业务解释，原因码应由文档和后台字典解释。
- 不在前端页面组件里解释服务端裁决逻辑。
- 不用 TODO 代替阶段边界；若暂不实现，应写入实施路线，而不是留在代码里。

当前拼课 explain 的来源是：

```text
SceneCandidateLoader
-> PrimaryOfferResolver
-> CourseGroupSceneExplainService
-> ProductCardViewBuilder
-> ClientSceneProductCardVo
```

解释范围：

- 同一商品是否有可加入团队。
- 当前会员是否已经在团内。
- 团队是否已满、已成团、已补位成团、上课中、已结束、已关闭。
- 无可加入团队时提示可先开团。

不解释的范围：

- 不解释完整推荐算法，因为当前没有模型召回 / 粗排 / 精排。
- 不把排课地址缺失解释成库存不足。
- 不把服务容量等同于实物库存。
- 不向 C 端暴露内部风控规则和完整团队审计。

这一步对 P1 的补强点在于：`traceId` 仍然负责反查，`courseGroupJoinExplain` 负责把“本次为什么展示这个拼课状态”直接带到 C 端卡片上，避免小程序根据团队状态二次猜测。

# 18. 2026-04-28 Trace 诊断入口与后台面板回写

S10 已补第一版只读诊断入口：

```text
GET /api/marketing/resolution/diagnostics/trace?traceId=...&days=...
```

返回内容：

- 最近 1-7 天内的 scene resolve trace detail。
- 最近 1-7 天内的 cache invalidation trace detail。
- cache invalidation 命中的事件目录项。
- 与 traceId 相关的 incident 列表。

admin-web 已接入现有排障工单页面：

- `apps/admin-web/src/service/api/marketing/resolution.ts`
  - `fetchResolutionTraceDiagnostic` 使用 OpenAPI 生成类型，不新增前端同义 DTO。
- `apps/admin-web/src/views/marketing/resolution/incident/modules/trace-diagnostic-panel.vue`
  - 支持按 `traceId + days` 查询。
  - 展示场景裁决样本、缓存失效样本、关联事件目录和关联工单。
- `apps/admin-web/src/views/marketing/resolution/incident/index.vue`
  - 工单表增加 Trace 列。
  - 有 Trace 的工单可直接触发诊断面板查询。

当前策略仍是基础架构档：

- 不新增 Prisma model。
- 不新增 migration。
- 不保存完整商品快照。
- 不在 C 端暴露内部诊断。
- 不新增 admin 菜单、路由或权限 seed。

这一步能解释的问题：

- 这个 traceId 是否有场景裁决记录。
- 当时走了哪个 scene、releaseNo、channel。
- 本次裁决成功还是失败、耗时多少、是否有空模块。
- 同 trace 是否触发过缓存失效，触发事件是什么。
- 是否已有 incident 与该 trace 相关。

这一步不能解释的问题：

- 不能复原完整商品列表。
- 不能证明某个用户完整权益状态。
- 不能替代模拟器。
- 不能替代后续独立 trace 表。

后台诊断面板只消费诊断接口，不直接拼 Redis key。面板查询是只读动作，不触发重新裁决，也不改变线上运行态。
