# 1. 移交目的

本文档供**其他模型或工程师**接续实施：**不做新旧共存**，对外只保留一套营销主路径；数据层采用 **B 方案**（`CampaignDraft` / `CampaignRelease` 新表或等价命名）；**权益池使用独立表**，与活动草稿通过关联表绑定。

承接方实施前请先阅读：

1. 当前代码与 OpenAPI/generated types，确认 `/admin/activity`、Campaign、Entitlement 与 miniapp-client 的真实状态。
2. `docs/design/marketing-full-revamp/14-20` 专项边界文档。
3. `apps/backend/AGENTS.md`、`apps/backend/prisma/schema.prisma` 中 **`MktActivity`** 及相关 `mkt_activity_*` 表。

早期 `07-12` 阶段性方案已按过程稿删除；本文只保留仍可能影响迁移、回滚和旧表处理的交接口径。

---

# 2. 已锁定决策（勿再争论范围）

| 项                 | 决策                                                                                                                                                                                                           |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 共存策略           | **不允许**新旧 admin 主路径长期并存；**删除**对外 `admin/activity`（或等价：不再注册 HTTP，仅允许内部迁移代码读取旧表一次）                                                                                    |
| 数据模型           | **B**：新建 **草稿 / 发布** 两张（或一套）表作为营销编排事实源；**不采用**「仅改 URL、表仍为唯一 `MktActivity`」的 A 方案                                                                                      |
| 权益池             | **独立表** + 与 `CampaignDraft` 的**显式关联**（多对多或一对多按产品二选一，默认推荐多对多：`draft` ↔ `pool` 关联表）；`Release` 建议存 **pool 引用 + 发布时快照**（JSON 或子表），避免发布后改池影响已发版本 |
| 现有 `MktActivity` | 迁移后：**不再作为 admin 写入入口**；表可保留只读归档、或改名废弃、或最终 DROP（由你们数据保留策略决定，须在迁移章节写死）                                                                                     |

# 3. 目标架构（交接后应达到）

## 3.1 后端 HTTP（admin）

- **唯一**活动编排入口：`/admin/marketing/campaigns/*`（列表、详情、创建、复制、归档、停止、分步 PATCH、预检、发布等，按蓝图分批实现，但**路由前缀不重复**）
- **权益池**：`/admin/marketing/entitlement/pools/*`（CRUD + 分页列表 + 搜索）+ 保留现有 **`definition`**、**`compile`**（路径可继续为 `/admin/marketing/entitlement/definition` 与 `.../compile`）
- **删除**：`ActivityAdminController` 对外暴露的 **`/admin/activity/*`**（实现前先全仓 grep 调用方：定时任务、其它模块、OpenAPI 客户端）

## 3.2 admin-web

- 路由与菜单：**只保留** `/marketing/campaigns/*`（及权益池 `/marketing/entitlement-pool` 等子页）；**移除** `/marketing/activity/*` 或永久重定向至 campaigns（二选一，推荐删除 + 更新书签文档）
- API：`activity.ts` 中面向 **`/admin/activity`** 的封装 **删除或改为** 调用新 `campaign` API；**权益池页**去掉 `localStorage`，列表与搜索全部走服务端

## 3.3 协议与壳子

- `GET /marketing/protocol/definition`：可保留为**静态契约**；若要做运行时校验，在待办中单独立项
- 现有 **`CampaignShellService`** 的静态 steps/tabs：**被真实 DB + 服务替代**，壳接口路径可保留，**响应不得再依赖占位 `cmp_shell_preview` 语义**

# 4. Prisma / 表设计建议草案（实施时按项目命名规范微调）

> 表名仅为建议，与现有 `mkt_*` 风格对齐即可。

## 4.1 `MktCampaignDraft`（草稿）

- `id`, `tenantId`, `name`, `type`（或枚举）, `status`（DRAFT / PENDING_APPROVAL / …）
- 分步字段：**要么**宽 JSON 多列（`foundationJson`, `audienceJson`, …）**要么**单 `payloadJson` + 版本号；须与向导 7 步、工作台 Tab 写入边界一致
- `ownerUserId`, `createdBy`, `updatedBy`, `createTime`, `updateTime`
- 可选：`sourceActivityId`（一次性迁移时记录旧 `MktActivity.id`）

## 4.2 `MktCampaignRelease`（发布）

- `id`, `tenantId`, `draftId`, `version` / `releaseNo`, `status`, `publishedAt`, 快照字段（整包 JSON 或规范化列）
- **权益绑定**：`entitlementSnapshot`（JSON）或 `MktCampaignReleaseEntitlementPool` 子表，存 **poolId + 编译快照**

## 4.3 `MktEntitlementPool`（独立表）

- `id`, `tenantId`, `name`, `poolType`（PRODUCT | COUPON | POINTS）
- 来源字段：`sourceType`, `sourceKey`, `memberId`, `templateId`, `taskId` 等（与现有 `CompileEntitlementPool` DTO 对齐）
- `touchpoints`（JSON 数组或关联表）、`status`（DRAFT | COMPILED | FAILED）
- 可选：`lastCompiledAt`, `compilePreviewJson`, `riskSummaryJson`, `compileArtifactsJson`
- 审计字段

## 4.4 关联表 `MktCampaignDraftEntitlementPool`

- `draftId`, `poolId`, `sortOrder`, `createTime`
- 唯一约束按产品：`(draftId, poolId)` 等

## 4.5 与现有 `MktActivity` 的差异提醒

当前 `MktActivity` 存在 `@@unique([tenantId, type])`：**同一租户同一 type 仅一条**。新 Campaign 若允许多个同 type 活动，**不可**照搬该唯一约束到新表；迁移时需决定：

- 是否废弃「每租户每 type 唯一」产品规则；或
- 迁移后仍用 `type` 做业务唯一，则在 Draft 表保留等价唯一索引。

# 5. 数据迁移（必写进待办）

1. **分析**：`MktActivity`、`MktActivityTouchpoint`、`MktActivityParticipation` 与审批、场景、resolution 的引用链（grep `activityId`、`MktActivity`）
2. **脚本**：`Activity` → `CampaignDraft` 初始行 + 可选首条 `CampaignRelease`；`rewards` / 规则 JSON 中若含权益语义，映射到 **`MktEntitlementPool` + 关联表** 或写入 `draft` JSON（由产品决定）
3. **验证**：迁移后行数、租户隔离、关键 type 抽样对比
4. **切换**：admin 只读新表 → 删旧 HTTP → 最后阶段再考虑是否 DROP `mkt_activity`（强烈建议分 PR：先停写再删表）

---

# 6. 相关风险与依赖（交接模型必读）

1. **小程序 / client**：若存在依赖旧活动 ID 或旧 client 接口，须与 admin **同一迭代**改契约，否则断裂。
2. **权限**：种子与 `RequirePermission` 从 `activity:*` 迁到 `marketing:campaign:*`（或你们统一前缀），并更新角色绑定。
3. **缓存**：`MarketingCacheInterceptor` 与各模块 cache key 是否绑定旧 path/id，切换后一并清理或重绑。
4. **OpenAPI / 类型**：`pnpm build:backend` + `pnpm generate-types` + admin 类型修复为流水线一环。
5. **`@@unique([tenantId, type])`**：见 §4.5，避免迁移后无法创建多条同 type 草稿。

# 9. 验收口径（最小）

1. admin 侧 **不存在**可调用的 **`/admin/activity` 列表/CRUD**（或项目约定的等价旧路径）。
2. 活动列表/搜索/日历/驾驶舱（若保留功能）**全部**基于 **CampaignDraft/Release** 新数据源。
3. 权益池列表/搜索/编辑 **全部**基于 **`MktEntitlementPool`** 服务端 API，**无** `localStorage` 业务数据。
4. 租户隔离与权限与迁移前 **等价或更强**。
5. CI：`backend` + `admin-web` 相关测试通过。
