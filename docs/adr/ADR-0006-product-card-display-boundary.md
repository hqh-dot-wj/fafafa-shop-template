# ADR-0006：商品卡展示标签与营销价签边界

## 背景

用户端商品卡同时承载商品信息、活动价格、场景入口和下一步动作。历史实现容易把场景名、活动价签、商品运营标签混在同一个展示位置，导致用户误解，也让前端在没有稳定契约时猜测“新品、爆品、自营、包邮”等语义。

## 决策

1. **商品卡展示模型独立**：miniapp 分类页、商品列表页、首页精选区统一消费商品卡展示模型，不在页面内各自拼活动和商品信息。
2. **营销价签只属于价格区**：`primaryOffer.tagLabel`、活动价和 `activityContextKey` 只表达营销权益，不作为商品属性标签展示。
3. **商品展示标签只来自后端投影**：`displayTags` 表达商品展示事实或运营展示标签，不参与价格计算、营销裁决、下单活动锁定。
4. **场景和模块只做来源归因**：`sceneCode`、`moduleCode`、`entrySource` 不展示成商品标签，只用于跳转、归因和统计。
5. **标签后台维护暂缓**：在标签业务口径、运营维护需求、批量能力和租户隔离模型稳定前，不新增标签定义表、标签关联表或后台标签管理页。

## 后果

- 商品卡第一屏保持“少而准”：商品名、图片、价格/营销价签、必要说明和一个主动作。
- 前端不得再根据零散字段自行推断新品、爆品、自营、售罄等标签。
- `isHot` 当前只能映射为“门店推荐”，不得展示为“爆品”，因为它没有销量事实支撑。
- 后续若启动后台标签维护，必须作为新的跨 app 高风险改动处理，覆盖 Prisma、权限、审计、批量打标、OpenAPI/common-types 和 C 端展示验证。

## 后台标签维护启动条件

只有同时满足以下条件，才重新评估后台标签维护：

- 商品卡展示模型和 `displayTags` 契约已稳定。
- 自营、爆品、限量等标签的事实来源已经确认。
- 运营明确需要人工维护标签，且规则自动生成不能覆盖。
- 批量打标、导入导出、筛选和审计诉求明确。

未来推荐方向是“标签定义表 + 商品关联表”，而不是把人工标签塞进门店商品 JSON。该方案能承载启停、排序、批量、筛选和审计，但会触发 Prisma migration、权限、OpenAPI/common-types、admin-web 页面和 C 端展示验证，必须单独走高风险确认。
