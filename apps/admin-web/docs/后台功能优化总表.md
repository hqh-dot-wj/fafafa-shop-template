# 后台功能优化总表（真实约束版）

> 本表不是按“灵感清单”罗列，而是按 **真实代码基础、当前架构边界、实施价值、实现成本** 重新收敛。  
> 目标不是把后台做成概念上的“大运营平台”，而是优先补齐最影响高频运营体验、且当前项目真的适合推进的能力。

## 一、判断口径

### 1. 评估公式

`实现价值 = 使用频次 x 当前痛点 x 复用范围 x 可落地性`

其中“可落地性”重点看四件事：

1. 前后端是否已有能力雏形
2. 是否需要新增跨模块底层模型
3. 是否撞上多租户/权限/异步任务边界
4. 是否会进入高风险域（资金、支付、认证）

### 2. 四种结论

| 结论     | 含义                               | 处理方式                     |
| -------- | ---------------------------------- | ---------------------------- |
| **保留** | 方向正确，且已有基础或收益足够高   | 进入排期，优先做闭环         |
| **下调** | 有价值，但不是现在最值得投入的点   | 放到第二波，等待前置条件成熟 |
| **删除** | 现阶段性价比低，或概念大于实际价值 | 不纳入近期规划               |
| **新增** | 从真实代码出发发现的更高杠杆项     | 优先级可高于原始灵感         |

## 二、总览结论

| 分类     | 项目                                                                                            |
| -------- | ----------------------------------------------------------------------------------------------- |
| **保留** | SKU 矩阵编辑与继承、订单处理台深化、导入导出标准化、工作台深化、租户同步收敛版                  |
| **下调** | 会员分群与批量运营、营销复制/冲突检测、财务解释与联查增强、库存异常视图                         |
| **删除** | 统一异常中心、统一业务任务中心、全对象租户同步平台、全域版本回滚                                |
| **新增** | 派单效率优化、批量结果协议统一、保存筛选方案/常用视图、财务回跳业务源单、模拟能力就地嵌入配置页 |

## 三、保留：应该继续做，但按真实基础收口

| 项目                                        | 结论 | 依据                                                                                                                                                                     | 优先级 |
| ------------------------------------------- | ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------ |
| **SKU 矩阵编辑器 + 继承/默认值 + 变更预览** | 保留 | `pms/global-product-create` 已有分步创建与 `Step3Sku.vue`，但 `PMS` 后端仍以单条 CRUD 为主，说明这里是真缺口，不是前端没补而已                                           | **P0** |
| **订单处理台深化**                          | 保留 | `store/order/list` 已有批量核销、批量退款；`store/order/detail` 已能查业务操作日志并跳活动审计、佣金审计，说明方向对，但仍停在“可操作”，还没到“高频处理台”               | **P0** |
| **导入导出标准化**                          | 保留 | `store/product` 已有批量导入、批量调价；库存、订单、会员均已有导出入口，但协议和回执不统一，最适合先做标准化而非重新造平台                                               | **P0** |
| **工作台深化**                              | 保留 | `home/modules/quick-actions.vue` 已有待处理订单、待审核提现、待审核升级与快捷入口，不是从零开始，适合继续补“最近任务/失败提醒/常用入口”                                  | **P0** |
| **租户同步收敛版**                          | 保留 | 已有 `syncTenantDict`、`syncTenantConfig`、`syncTenantPackage`，但目前更像后台管理动作，不是成熟同步系统，适合做“已有对象的差异化和回执增强”，不适合直接泛化成全对象平台 | **P0** |

### 说明

- **订单中心不是缺批量，而是批量还不够强。** 后端 `store-order.service.ts` 的 `batchVerify`、`batchRefund` 目前是逐单循环处理，适合小批量，不适合大批量高峰。
- **PMS 商品域是真正薄弱点。** 前端 `fetchBatchDeleteGlobalProduct` 仍是循环单删，说明 `PMS` 这块的“批量能力”不是产品没想清楚，而是后端基础确实没长出来。
- **租户同步必须收口。** `syncTenantPackage` 里套餐菜单同步逻辑仍是注释占位，说明这块现在离“成熟同步体系”还差一层核心实现。

## 四、下调：有价值，但不应放在第一波

| 项目                               | 结论 | 原因                                                                                                                                                         | 建议阶段 |
| ---------------------------------- | ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------- |
| **会员分群与批量运营**             | 下调 | 会员页已有详情抽屉，能联查订单、流水、积分、操作日志，先把现有联查与批量动作补齐，比先上复杂分群引擎更值                                                     | **P1**   |
| **营销复制 / 冲突检测 / 版本治理** | 下调 | `marketing/resolution/simulator` 与 `resolution/priority` 已说明项目有“规则解释”基础，但营销体系仍在成长态，先做就地模拟与复制即可，不宜一口气上完整版本治理 | **P1**   |
| **财务解释与联查增强**             | 下调 | 当前财务侧已有提现、台账、佣金审核、看板，先补“回跳订单/会员/分销源单”比做完整解释引擎更划算                                                                 | **P1**   |
| **库存异常视图 / 调拨视图**        | 下调 | `store/stock` 已有批量调整，属于“已经能操作”；库存不是当前最痛点中心，优先级低于商品和订单                                                                   | **P1**   |
| **角色/字典/菜单模板化复制**       | 下调 | 有实施价值，但主要服务运维与交付，不是高频运营主链路，且容易和租户同步范围重复                                                                               | **P2**   |

## 五、删除：现阶段不建议纳入近期规划

| 项目                           | 结论 | 删除原因                                                                                                                   |
| ------------------------------ | ---- | -------------------------------------------------------------------------------------------------------------------------- |
| **统一异常中心**               | 删除 | 当前异常分散在订单、财务、调度日志、风控、钱包等域，没有统一异常模型。现在做大聚合页，很容易只是“把几个列表拼一起”         |
| **统一业务任务中心**           | 删除 | 项目里同时存在 `@Task + SysJob` 的调度任务和 Bull 队列异步任务，两套模型未统一；强行先做中心会混淆技术任务与业务任务       |
| **全对象租户同步平台**         | 删除 | 当前租户同步只适合字典、配置、套餐级别；若直接扩到商品、营销、支付、菜单、装修、素材等，会立即撞上权限、归属与冲突策略问题 |
| **全域版本回滚**               | 删除 | 资金、订单、租户配置都不是适合轻易承诺“回滚平台”的域，当前更该做前置预览、差异确认、回执和审计                             |
| **首页大改成统一运营中台门户** | 删除 | `home` 已有工作台与待办，不值得因为“看起来不够高级”而重做门户层；应在现有基础上深化                                        |

## 六、新增：从项目真实代码里看到的更高杠杆项

| 新增项                      | 原因                                                                                                   | 优先级 |
| --------------------------- | ------------------------------------------------------------------------------------------------------ | ------ |
| **派单效率优化**            | `store/order/dispatch` 方向对，但改派仍偏“输入 workerId”式操作，实际运营痛点比想象中的“异常中心”更直接 | **P0** |
| **批量结果协议统一**        | 订单、库存、商品批量接口都在返回成功/失败，但结构不统一；统一协议能直接提升前端复用和错误反馈质量      | **P0** |
| **保存筛选方案 / 常用视图** | 当前后台列表很多，订单、商品、提现、会员、库存都适合这个能力，投入远小于重做报表，收益却很高           | **P1** |
| **财务页回跳业务源单**      | 现在订单详情能跳财务审计，但反向链路弱；财务审核、台账、佣金页应支持一键打开订单、会员、分销来源       | **P1** |
| **模拟能力就地嵌入配置页**  | 已有 `marketing/resolution/simulator`，更应该把这种能力嵌到高风险配置页，而不是独立发展成更大平台      | **P1** |

## 七、模块级真实判断

### `home`

- **保留**：工作台深化
- **删除**：大改成全新中台首页
- **依据**：已有待办和快捷入口，缺的是“更贴近作业”的收口，而不是视觉层重做

### `pms`

- **保留**：SKU 矩阵编辑、批量字段、继承默认值、保存前 diff
- **新增**：导出后修改再导入的闭环
- **依据**：这里是当前最薄弱、最值钱的商品运营能力缺口

### `store/order`

- **保留**：批量处理深化、详情联查深化
- **新增**：派单效率优化
- **删除**：先做统一异常平台
- **依据**：已有批量核销/退款和详情联查，最该做的是把现有能力做强

### `store/product` / `store/product_market`

- **保留**：批量调价、批量导入、批量状态和结果回执统一
- **依据**：后端已经有 `batchImportProducts`、`batchUpdateProductPrice`，适合扩面，不适合推翻重来

### `store/stock`

- **下调**：异常中心、调拨大视图
- **保留**：批量结果与原因记录增强
- **依据**：库存已可批量调整，但不是当前第一痛点域

### `store/finance`

- **下调**：完整解释引擎
- **新增**：回跳业务源单、驳回原因模板、常用筛选
- **依据**：财务是高风险域，先提效，不先做重平台

### `marketing`

- **下调**：完整版本管理与跨租户复制
- **保留**：复制活动、规则模拟、冲突提示
- **新增**：把模拟能力嵌入具体配置流程
- **依据**：项目已具备模拟器雏形，更适合局部深化

### `member`

- **下调**：先做复杂分群平台
- **保留**：批量等级、积分、标签、发券等运营动作
- **依据**：详情抽屉已经具备多域联查，先把运营动作接上

### `system/tenant`

- **保留**：租户同步的差异预览、回执、失败项展示
- **删除**：立刻发展成全对象租户同步平台
- **依据**：已有同步接口，但核心同步边界还不稳定，尤其套餐菜单同步未闭环

### `monitor`

- **删除**：把它直接扩成业务任务中心
- **依据**：它现在更适合继续承担“技术任务/调度日志”，不应混入业务批处理任务概念

## 八、建议排期

### P0

1. `PMS` 的 SKU 矩阵编辑、继承默认值、变更预览
2. 订单处理台深化：批量备注、批量状态流转、派单效率、导出增强
3. 导入导出标准化：统一成功/失败回执、错误明细、再导入修正
4. 工作台深化：最近任务、失败提醒、常用入口
5. 租户同步收敛版：差异预览、回执、失败项，不扩对象范围

### P1

1. 会员批量运营动作
2. 营销复制、冲突提示、配置页内模拟
3. 财务回跳业务源单
4. 保存筛选方案 / 常用视图

### P2

1. 角色/菜单/字典模板化复制
2. 库存异常视图
3. 更复杂的营销版本治理

## 九、实施约束

1. **不要把“前端功能规划”建立在不存在的后端批接口上。** `PMS` 当前批量能力尤其薄。
2. **不要把“统一任务中心”作为第一阶段目标。** 现有 `monitor/job` 与 Bull 队列不是一套模型。
3. **租户同步只能做收敛，不要先做扩张。** 先把已有 `dict/config/package` 同步做稳。
4. **高风险域只做提效，不做冒进平台化。** 财务、支付、认证相关能力优先考虑可控与可审计。
5. **优先做“已有雏形的闭环增强”。** 这比另起一层平台，风险更小、收益更快。

---

## 十、结论

当前项目最值得推进的，不是“补齐你能想到的所有运营平台能力”，而是：

- 把 **商品/SKU、订单、导入导出、工作台、租户同步** 这几条主链路做深
- 把已经存在的雏形能力真正做成高频可用的工具
- 避免过早进入 **统一异常中心 / 统一业务任务中心 / 全对象同步平台** 这类看起来高级、实际上落地成本很高的方向

这版总表可直接作为产品排期底稿，后续若进入实现阶段，应再拆成：

1. 前端交互项
2. 后端接口项
3. 异步与审计项
4. 权限与租户边界项

| **财务回跳业务源单**              | 在台账、提现、佣金页增加订单/会员/分销来源跳转             | 返回关联主键与来源类型                                        | 需要梳理不同财务记录的来源映射                   | **P1**   | **M**  |
| **营销配置页内模拟**              | 在规则配置页直接带入当前表单参数模拟                       | 复用现有 `resolution/simulator` 能力，补更贴近页面的入参      | 容易做成重复页面；应优先复用                     | **P1**   | **M**  |

### 11.1 推荐启动顺序

1. **先做“批量结果协议统一 + 工作台深化 + 订单批量增强”**  
   原因：依赖较少，能最快体现运营效率提升。
2. **再做 `PMS` 的 SKU 矩阵与批量能力**  
   原因：价值最高，但对后端接口和保存模型要求也最高。
3. **随后推进租户同步收敛版**  
   原因：平台价值高，但必须建立在边界清晰基础上。
4. **最后做会员、营销、财务的第二波提效项**  
   原因：这些方向更适合建立在前面标准化能力已经形成之后。

### 11.2 拆任务建议

每个需求项建议统一拆成 4 类子任务：

1. **交互任务**：页面、弹窗、回执、表格、筛选、跳转
2. **接口任务**：批量接口、导入导出、校验结果、聚合统计
3. **审计任务**：操作日志、失败原因、结果回执
4. **边界任务**：权限、多租户、幂等、异步与超时

### 11.3 不建议单独立项的内容

- “统一异常中心”不要单独开项目，应拆进订单、库存、财务、同步的失败回执与异常筛选里
- “统一业务任务中心”不要先开，等业务批处理模型稳定后再考虑是否需要统一入口
- “全对象租户同步平台”不要先开，先把 `dict/config/package` 三类同步做稳
