# Miniapp Page Change

## 目标

为 miniapp-client 的页面、store、API、布局和双端差异处理提供完整执行手册，重点避免以下重复错误：

1. H5 / MP 差异没有显式边界，靠猜测兼容
2. 用户可见状态语义散落在页面和组件内，直接展示原始 code 或 `statusText`
3. 页面层直接堆请求、状态与提交流程，没有通过 API / store 收敛
4. 高风险链路只验证 happy path，没有验证回跳、刷新、失败和重进语义
5. 布局问题没修容器链就直接改业务逻辑

## 进入本手册前

先读根 `AGENTS.md` + `apps/miniapp-client/AGENTS.md`，再读本手册。涉及验证再读 `.codex/playbooks/verification-gates.md`；涉及跨端语义/字典再读 `.codex/playbooks/dict-and-job-change.md`。本手册只补 miniapp 页面改动时双端、布局、交互流程判断。

## miniapp 改动分类

先判断当前属于哪类：

- 页面 UI 改动
- 页面逻辑改动
- `src/api/*` 改动
- `src/store/*` 改动
- 布局 / 条件编译 / tabbar 改动
- 登录、下单、支付、分销等高风险链路改动
- 跨 app 契约改动

## 默认顺序

1. 先确定页面类型与高风险级别
2. 再确认影响平台：H5、MP 或双端
3. 若是布局问题，先修容器链、滚动链和安全区
4. 再确认 API、store、用户可见语义与提交路径
5. 最后补空态、错误态、加载态与行为验证

## 输出协议与分析要求

按 `docs/governance/AGENT_OUTPUT_PROTOCOL.md` 输出。miniapp 场景额外覆盖：Mermaid 页面流程图 + H5 / MP 双端边界。

## 必查事项

### 平台边界与布局链

必须确认：

- 当前问题发生在 H5、MP 还是双端
- 是否需要 `disableScroll: true`
- 是否使用“固定头尾 + 中间 `scroll-view`”
- H5 下是否需要固定根容器防止整页滚动
- 弹层、分享、tabbar、安全区是否存在平台差异

禁止：

- 用运行时平台判断代替应当使用的条件编译
- 页面根容器、`scroll-view`、安全区链路没理顺，就先去改业务逻辑
- 只在一个端上试过，就默认另一个端也没问题

### API、类型与 store 边界

必须确认：

- API 类型是否优先来自 `@libs/common-types`
- 页面是否直接堆请求与业务状态，而不是通过 `src/api/` 和 `src/store/` 收敛
- 是否存在本地手写重复契约
- backend 契约变更是否已经同步到 miniapp 端映射

允许本地类型的场景：

- 纯页面 view-model
- 纯组件内部展示态
- 临时兼容字段映射层

禁止：

- 页面里直接散落 request 实现
- 把整页对象直接拿去提交订单、支付、升级申请等写接口
- 使用 `as any` 或 `@ts-ignore` 掩盖契约问题

### 用户可见语义与文案

以下都属于必须统一治理的用户可见语义：

- 团状态
- 支付状态
- 成员角色
- 订单状态
- 升级申请状态
- 分销等级或团队身份

默认规则：

1. 用户可见状态必须来自统一映射层，不能在多个页面和组件各写一套
2. 若后端返回 `statusText`、`teamStatusText`、`payStatusText` 等字段，必须确认其为稳定可展示语义
3. 同一语义在列表、详情、卡片、进度组件、分享文案、结果页中必须一致
4. 禁止把英文 code、数据库原值或临时 fallback 直接展示给用户

禁止：

- 页面和子组件分别维护两套同义状态映射
- 因为“用户看不懂代码”就直接在页面里临时拼一套中文文本，却不治理来源
- 登录、支付、结果页保留临时英文提示语

### 高风险链路

以下链路必须提升验证等级：

- 登录注册
- 下单确认
- 支付拉起与支付结果
- 升级申请
- 分销团队与角色切换

这些链路至少要检查：

- token / 用户态写入与刷新
- 回跳路径
- 页面重进语义
- 失败提示
- 状态刷新时机
- 双端差异

### 批量语义与列表操作

若页面存在批量删除、批量操作、批量管理：

- 优先确认 backend 是否已有批量接口
- 不要在页面中循环调用 N 次单接口
- 若确实只能逐条执行，必须说明原因，并处理部分失败反馈

## 测试与验证要求

### 不足以证明正确的验证

以下方式不能单独作为完成证据：

- 只跑 `lint:h5` / `typecheck:h5`
- 只验证 API 请求构造，不验证页面行为
- 只在 H5 或只在 MP 单端验证，却宣称双端都通过
- 只验证 happy path，不验证登录失效、支付失败、订单状态刷新、回跳或页面重进

### 本次改动至少要覆盖什么

普通页面 / store / api：

- 至少确认加载态、空态、错误态与用户主流程

布局与滚动问题：

- 至少验证容器链、滚动链、安全区和弹层滚动行为

高风险链路：

- 必须验证成功路径、失败路径和重进/回跳语义
- 若声明影响双端，必须分别说明 H5 与 MP 的验证结论

### 建议验证命令

```powershell
pnpm lint:h5
pnpm typecheck:h5
```

若用户要求或当前改动属于高风险链路，还应补：

- 对应端的人工流程验证
- 必要时补自动化测试或等价的端到端行为验证

## 交付前自查

- 是否明确了当前问题影响 H5、MP 还是双端
- 是否先修了布局链，而不是直接动业务逻辑
- 是否统一了用户可见状态语义
- 是否仍存在原始 code、英文状态或多个组件各写一套映射
- 是否把请求、状态、提交流程收敛到了 API / store
- 是否覆盖了空态、错误态、加载态和高风险失败路径

## miniapp 常见返工点

- 只在 H5 测过，就默认 MP 也正常
- 布局容器链没理顺，靠业务逻辑补救滚动问题
- 详情页、进度组件、结果页分别维护不同状态文案
- 页面直接调用 request，绕开 `src/api/` 与 `src/store/`
- 高风险链路只验证成功路径，没有验证回跳和重进语义
