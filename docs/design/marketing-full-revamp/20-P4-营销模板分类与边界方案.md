P4 要回答的问题是：

```text
我们项目里的“营销模板”能不能理解为文章里的“玩法模板”？
优惠券、积分、营销模板、场景卡片、拼课模板分别是什么？
是否需要把现有表硬拆成一套新的模板体系？
后续新增玩法时，应该沿用什么边界？
```

本文只做实施分析和边界设计，不直接修改 Prisma schema、seed 或前后端代码。

# 2. 核心结论

当前项目里的“营销模板”不能简单等同于文章里的“玩法模板”。

更准确的说法是：

```text
项目里已经存在一组模板能力，
其中 PlayTemplate + PLAY_REGISTRY + StorePlayConfig 的组合，才接近文章里的玩法模板。
```

但项目里至少有五类模板：

| 类型            | 当前承载                                                                    | 说明                                               |
| --------------- | --------------------------------------------------------------------------- | -------------------------------------------------- |
| 玩法定义模板    | `PLAY_REGISTRY`、`@PlayStrategy`、玩法 DTO                                  | 真正决定怎么玩、是否有实例、是否有状态、是否可失败 |
| 玩法配置模板    | `PlayTemplate`                                                              | 当前偏动态表单与展示外壳，不能单独执行             |
| 门店玩法配置    | `StorePlayConfig`                                                           | 租户把某个玩法应用到某个商品 / 服务上的实例化配置  |
| 权益模板        | `MktCouponTemplate`、`MktPointsRule`、`MktPointsTask`、`MktEntitlementPool` | 决定发什么、用什么、预算或库存是什么               |
| 展示 / 场景模板 | `MktPolicy(policyType = CARD_TEMPLATE)`、`MktScene`、`MktSceneModule`       | 决定怎么展示、在哪展示、按什么策略出数             |

因此 P4 不建议先硬拆表。

推荐做法是：

```text
先冻结模板分类和代码目录，
再调整后台心智和 API 边界，
最后才评估是否补 templateCategory 等字段。
```

# 3. 当前事实

## 3.1 PlayTemplate 当前更像表单模板

当前 Prisma 模型：

```text
PlayTemplate
  code
  name
  ruleSchema
  unitName
  uiComponentId
  productId / skuId / productName
```

对应代码：

- `apps/backend/prisma/models/80-marketing.prisma`
- `apps/backend/src/module/marketing/template/template.service.ts`
- `apps/backend/src/module/marketing/template/dto/template.dto.ts`
- `apps/backend/src/module/marketing/template/vo/template.vo.ts`
- `apps/admin-web/src/views/marketing/template/index.vue`
- `apps/admin-web/src/service/api/marketing/template.ts`

当前 `PlayTemplateService.create` 会生成 `PT_xxx` 形式的模板编码。

这带来一个重要问题：

```text
如果这个 PT_xxx 模板没有对应 PLAY_REGISTRY 和 @PlayStrategy，
它不能被 StorePlayConfig 当成可执行玩法使用。
```

所以 `PlayTemplate` 当前不是完整玩法定义，它更像：

```text
玩法配置表单模板 + UI 组件映射 + 运营可见名称
```

## 3.2 PLAY_REGISTRY 才是真正可执行玩法目录

当前可执行玩法定义在：

- `apps/backend/src/module/marketing/play/play.registry.ts`
- `apps/backend/src/module/marketing/play/play-strategy.decorator.ts`
- `apps/backend/src/module/marketing/play/play.factory.ts`
- `apps/backend/src/module/marketing/play/course-group-buy.service.ts`
- `apps/backend/src/module/marketing/play/flash-sale.service.ts`
- `apps/backend/src/module/marketing/play/member-upgrade.service.ts`

`PLAY_REGISTRY` 记录：

- `code`
- `name`
- `hasInstance`
- `hasState`
- `canFail`
- `canParallel`
- `ruleSchema`
- `defaultStockMode`
- `description`

这比 `PlayTemplate` 更接近玩法定义。

`@PlayStrategy('COURSE_GROUP_BUY')` 还会强校验 code 是否在 `PLAY_REGISTRY` 中注册。也就是说：

```text
新增可执行玩法，必须先经过代码注册。
```

这是正确的边界，不能把它开放给运营动态新增。

## 3.3 StorePlayConfig 是租户玩法实例化配置

当前 `StorePlayConfig` 包含：

- `tenantId`
- `serviceId`
- `serviceType`
- `templateCode`
- `rules`
- `rulesHistory`
- `stockMode`
- `aggregateEnabled`
- `zoneEnabled`
- `displayPriority`
- `commissionMode`
- `commissionRate`
- `status`

对应代码：

- `apps/backend/src/module/marketing/config/config.service.ts`
- `apps/backend/src/module/marketing/config/dto/config.dto.ts`
- `apps/backend/src/module/marketing/config/activity-conflict.matrix.ts`
- `apps/admin-web/src/views/marketing/config/index.vue`

当前创建逻辑已经做了几件关键事：

- 校验 `PlayTemplate` 是否存在。
- 通过 `PlayStrategyFactory.getStrategy(dto.templateCode)` 校验策略。
- 校验关联商品 / SKU 是否存在。
- 检查同商品活动互斥。
- 根据商品类型自动决定 `stockMode`：实物 `STRONG_LOCK`，非实物 `LAZY_CHECK`。

这说明项目已经有比较清晰的执行边界：

```text
PlayTemplate 提供模板记录；
PLAY_REGISTRY 提供技术能力；
StorePlayConfig 提供门店投放配置。
```

问题是后台页面和命名会让人以为“新增模板 = 新增玩法”，这需要纠正。

## 3.4 优惠券和积分不是玩法模板

优惠券模板：

- `MktCouponTemplate`
- `MktUserCoupon`
- `MktCouponUsage`

积分规则：

- `MktPointsRule`
- `MktPointsTask`
- `MktPointsTransaction`

权益池：

- `MktEntitlementPool`
- `MktCampaignDraftEntitlementPool`

这些对象回答的是：

```text
发什么？
怎么用？
库存 / 预算 / 有效期是什么？
```

它们不是“怎么玩”。

优惠券可以被玩法引用，也可以被场景投放引用；积分可以被任务、订单、签到、退款引用。它们是权益和资产，不应被强行归入玩法模板。

## 3.5 场景和策略不是玩法模板

当前场景链路：

- `MktScene`
- `MktSceneModule`
- `MktSceneRelease`
- `MktPolicy`

对应代码：

- `apps/backend/src/module/marketing/scene/scene.service.ts`
- `apps/backend/src/module/marketing/policy/policy.service.ts`
- `apps/backend/src/module/marketing/resolution/resolution.service.ts`
- `apps/admin-web/src/views/marketing/scene/**`
- `apps/admin-web/src/views/marketing/scene-module/**`
- `apps/admin-web/src/views/marketing/policy/**`

其中 `MktPolicy.policyType` 已经把策略分成：

- `SOURCE`
- `RESOLVER`
- `AUDIENCE`
- `SORT`
- `CARD_TEMPLATE`

`CARD_TEMPLATE` 是展示模板，不是玩法模板。

`MktSceneModule.cardTemplateCode` 当前通过发布前预检校验为 `CARD_TEMPLATE` 策略。这条线应该继续保留，不要和 `PlayTemplate.uiComponentId` 混成一个概念。

## 3.6 Campaign 是编排壳，不是模板本身

当前活动壳：

- `MktCampaignDraft`
- `MktCampaignRelease`
- `foundationJson`
- `audienceJson`
- `rightsJson`
- `stagesJson`
- `deliveryJson`
- `constraintsJson`

它回答的是：

```text
一次营销方案如何被组织、审核、发布和回滚。
```

Campaign 可以引用权益池、场景、触点、阶段，但它不是单个玩法模板。

# 4. 目标分类模型

P4 建议把“模板”拆成认知上的六类，不要求一开始就拆成六套表。

| 分类         | 英文名                    | 职责                               | 当前落点                                                   | 是否运营可新增 |
| ------------ | ------------------------- | ---------------------------------- | ---------------------------------------------------------- | -------------- |
| 玩法能力     | Play Capability           | 系统支持什么玩法、是否有实例和履约 | `PLAY_REGISTRY`、策略 Service                              | 否             |
| 玩法表单模板 | Play Config Template      | 玩法配置表单、字段、单位、组件映射 | `PlayTemplate`                                             | 受限           |
| 门店投放配置 | Store Play Config         | 某租户某商品启用某玩法和规则       | `StorePlayConfig`                                          | 是             |
| 权益模板     | Entitlement Template      | 券、积分、权益池、商品权益         | `MktCouponTemplate`、`MktPointsRule`、`MktEntitlementPool` | 是             |
| 展示模板     | Display Template          | 卡片、弹窗、详情组件、UI 配置      | `MktPolicy(CARD_TEMPLATE)`、`uiComponentId`                | 受限           |
| 场景编排模板 | Scene / Campaign Template | 页面模块、发布快照、活动编排       | `MktScene`、`MktCampaignDraft`                             | 是             |

这套分类的意义不是为了增加复杂度，而是避免下面几种混淆：

- 把优惠券模板误认为玩法模板。
- 把卡片模板误认为玩法模板。
- 把场景模块误认为玩法模板。
- 把运营新增表单误认为系统新增了可执行玩法。
- 把拼课的履约能力复制到所有玩法。

# 5. 和文章“玩法模板”的关系

如果文章里的“玩法模板”指的是：

```text
玩法规则 + 触发事件 + 奖励配置 + 状态变化 + 风控约束
```

那么项目中的对应关系是：

```text
PLAY_REGISTRY
  + PlayTemplate
  + StorePlayConfig
  + PlayStrategy Service
  + PlayInstance
  + 相关权益模板
```

而不是单独的 `PlayTemplate`。

因此可以这样理解：

```text
我们有玩法模板能力，但它分散在代码注册表、模板表、门店配置和策略 Service 里。
```

后续要汲取文章的长处，不是照搬一个大模板表，而是补上两件事：

1. 让每种模板的边界可解释。
2. 让玩法模板与权益、库存、履约、展示之间的关系可追踪。

# 6. 模块职责建议

## 6.1 marketing/play

职责：

- 可执行玩法白名单。
- 玩法技术能力元数据。
- 策略服务注册。
- 是否有实例、状态、失败、履约、库存模式。

不负责：

- 运营自由创建新玩法。
- 优惠券库存。
- 积分 lot。
- 场景卡片渲染。

建议补充的元数据：

```text
category: PLAY_CAPABILITY
resourceSemantics: PRODUCT / SERVICE / MEMBERSHIP / POINTS / COUPON
fulfillmentType: NONE / ORDER / COURSE / LOGISTICS / MEMBERSHIP
capacityType: STOCK / SERVICE_CAPACITY / SCHEDULE / BUDGET
riskLevel: LOW / MEDIUM / HIGH
```

## 6.2 marketing/template

职责：

- 维护玩法配置模板。
- 展示规则表单字段。
- 标记它绑定哪个 `registryCode`。
- 限制哪些模板可用于 `StorePlayConfig`。

不负责：

- 创建新的技术玩法。
- 定义权益库存。
- 定义场景卡片。

当前最需要纠正的是：

```text
后台“新增模板”不能让人误解为“新增可执行玩法”。
```

## 6.3 marketing/config

职责：

- 将玩法能力应用到具体商品 / 服务。
- 保存租户规则和历史版本。
- 决定上下架、聚合展示、专区展示、佣金。
- 做活动互斥校验。

不负责：

- 维护玩法能力目录。
- 维护券模板。
- 维护场景发布快照。

## 6.4 coupon / points / entitlement

职责：

- 提供可被玩法或活动引用的权益。
- 维护库存、预算、有效期、用户资产。
- 提供可解释的发放、锁定、核销、退款记录。

不负责：

- 判断“怎么玩”。
- 管理页面场景出数。

## 6.5 scene / policy / campaign

职责：

- 决定在哪里展示。
- 决定召回、受众、排序、裁决和卡片模板。
- 管理发布快照。
- 支撑 P1 的 trace 和 explain。

不负责：

- 新增可执行玩法。
- 处理订单退款。
- 处理积分资产账。

# 7. 数据模型影响

P4 不建议立即动表。

第一阶段可以只做代码目录和后台展示心智：

```text
TemplateCategoryCatalog
PlayCapabilityCatalog
DisplayTemplateCatalog
EntitlementTemplateCatalog
```

后续如果要进入 Prisma schema，可能补充：

```text
PlayTemplate
  templateCategory
  registryCode
  executionMode: REGISTRY_BACKED / CONFIG_ONLY / DISPLAY_ONLY
  resourceSemantics
  fulfillmentType
  capacityType
  status
```

但这属于 Prisma schema 改动，按仓库规则是高风险，需要单独确认。

优惠券、积分、权益池、场景策略不建议合并到 `PlayTemplate` 表。

原因是：

- 优惠券模板有库存和用户券状态机。
- 积分规则后续会发展为资产账和预算。
- 场景卡片模板属于展示策略。
- Campaign 发布需要快照和审核流。

它们可以被统一目录解释，但不应被强行合表。

# 8. 实施分析

## 8.1 后端第一步：代码目录

建议新增或整理一个只读目录：

```text
apps/backend/src/module/marketing/template/template-category.catalog.ts
```

目录内容用于解释：

```text
PLAY_CAPABILITY
PLAY_CONFIG_TEMPLATE
STORE_PLAY_CONFIG
ENTITLEMENT_TEMPLATE
DISPLAY_TEMPLATE
SCENE_ORCHESTRATION
CAMPAIGN_ORCHESTRATION
```

这个目录不驱动数据库写入，只提供后台展示、接口返回和测试断言。

## 8.2 PlayTemplate 创建边界

当前 `PlayTemplateService.create` 会自动生成 `PT_xxx` code。

如果该记录继续被称为“玩法模板”，会有歧义：

| 情况                                                         | 结果                         |
| ------------------------------------------------------------ | ---------------------------- |
| code 是 `COURSE_GROUP_BUY` / `FLASH_SALE` / `MEMBER_UPGRADE` | 可以绑定策略，接近可执行玩法 |
| code 是 `PT_xxx`                                             | 只是表单模板，不能直接执行   |

推荐调整后台语义：

```text
注册表玩法：来自 PLAY_REGISTRY，不允许运营直接新增技术 code。
表单模板：运营可维护字段、单位和 UI 组件，但必须绑定已有 registryCode 才能用于门店玩法配置。
```

后续代码可考虑：

- `CreatePlayTemplateDto` 支持选择 `registryCode`。
- `PlayTemplateService.create` 禁止创建 `executionMode = REGISTRY_BACKED` 但没有 `registryCode` 的模板。
- `StorePlayConfigService.create` 报错时区分“模板不存在”和“模板未绑定可执行策略”。

## 8.3 StorePlayConfig 选择器

当前 `apps/admin-web/src/views/marketing/config/index.vue` 从模板列表加载下拉选项。

后续应筛选：

```text
只展示可执行玩法模板。
```

也就是：

- 有对应 `PLAY_REGISTRY`。
- 有对应策略 Service。
- 状态可用。
- 不属于展示模板、权益模板或纯表单模板。

否则运营选了 `PT_xxx` 后，后端 `PlayStrategyFactory.getStrategy` 会失败，体验上像是“模板配置成功但活动不可用”。

## 8.4 展示模板与 cardTemplateCode

当前 `MktSceneModule.cardTemplateCode` 通过 `MarketingSceneService.collectPolicyRefs` 校验为 `CARD_TEMPLATE`。

这条线是正确的。

后续不要把 `cardTemplateCode` 改成引用 `PlayTemplate.code`。

推荐保持：

```text
玩法展示组件：PlayTemplate.uiComponentId
场景卡片模板：MktPolicy(policyType = CARD_TEMPLATE)
C 端渲染器：scene-module-renderer / marketing display hooks
```

三者可以建立映射，但不能合并概念。

## 8.5 权益模板边界

优惠券模板、积分规则、权益池后续都可以被 Campaign 或玩法引用。

但引用方式应该是：

```text
玩法 / 活动 -> entitlement reference
```

而不是：

```text
玩法模板表里直接塞券规则、积分规则和权益池 JSON
```

这样 P3 的积分资产账和优惠券库存不会被玩法模板拖乱。

## 8.6 新玩法接入清单

新增一个真正可执行玩法时，应按这个顺序：

```text
1. 在 PLAY_REGISTRY 中注册玩法能力。
2. 创建 RulesDto。
3. 创建 Strategy Service 并加 @PlayStrategy。
4. 补 PlayTemplate 或 seed，让后台有表单模板。
5. 补 activity-conflict.matrix。
6. 补 StorePlayConfig 创建校验。
7. 判断是否需要 PlayInstance。
8. 判断是否需要库存、服务容量、排课或履约。
9. 判断是否需要事件目录。
10. 判断是否需要场景出数解释。
11. 判断是否需要权益模板：券、积分、商品权益、课时。
12. 补 admin-web 配置页和 C 端展示组件。
```

拼课属于高复杂度样板；签到、领券、满减不应被迫复制拼课的排课和履约轴。

# 9. 阶段路线

## P4.1 模板分类冻结

目标：

- 冻结六类模板边界。
- 明确 `PlayTemplate` 不等于完整玩法。
- 明确优惠券和积分属于权益资产。

改哪些模块：

- 文档。
- 后续可补 TypeScript catalog。

不改哪些模块：

- 不改 Prisma。
- 不改 seed。
- 不改现有后台页面。

数据表影响：

- 无。

风险：

- 团队继续把“新增模板”理解成“新增玩法”。

验证方式：

- 用现有拼课、秒杀、会员升级、优惠券、积分、首页场景逐一映射到六类模板。

是否需要迁移历史数据：

- 否。

## P4.2 后端目录与 API 解释

目标：

- 让后端能返回模板分类和是否可执行。

改哪些模块：

- `apps/backend/src/module/marketing/play/play.registry.ts`
- `apps/backend/src/module/marketing/template/**`
- `apps/backend/src/module/marketing/config/config.service.ts`

不改哪些模块：

- 不改优惠券表。
- 不改积分表。
- 不改场景表。

数据表影响：

- 无；先用代码目录派生。

风险：

- 如果返回字段变更，需要走 OpenAPI 类型生成和前端适配。

验证方式：

- `COURSE_GROUP_BUY / FLASH_SALE / MEMBER_UPGRADE` 都能识别为可执行。
- `PT_xxx` 只能识别为表单模板或不可执行配置。
- `StorePlayConfig` 不能选择不可执行模板。

是否需要迁移历史数据：

- 否。

## P4.3 后台页面心智调整

目标：

- 后台区分“玩法能力”“玩法配置模板”“门店玩法配置”“展示模板”“权益模板”。

改哪些模块：

- `apps/admin-web/src/views/marketing/template/**`

# 12. 逻辑矫正

需要纠正几个判断：

1. “营销模板就是玩法模板”：不准确。项目里的模板分成玩法能力、表单模板、权益模板、展示模板、场景编排等多类。
2. “PlayTemplate 新增一条就是新增玩法”：不准确。没有 `PLAY_REGISTRY` 和 `@PlayStrategy`，它不能执行。
3. “优惠券和积分也是玩法模板”：不准确。它们是权益和资产，可以被玩法引用。
4. “先拆表最清楚”：不建议。当前更应该先冻结分类和后台心智，等查询和审计需求明确后再补字段。
5. “拼课模板经验可以复制给所有玩法”：不准确。拼课是 L4 复杂玩法样板，不能拖重领券、签到、满减。

# 13. 注释审查与注释方案

后续进入代码实现时，注释重点放在这些位置：

- `PLAY_REGISTRY`：说明这是技术能力目录，不是运营模板表。
- `PlayTemplateService.create`：说明运营创建的表单模板不能自动变成可执行玩法。
- `StorePlayConfigService.create`：说明为什么必须校验策略存在。
- `MarketingSceneService.collectPolicyRefs`：说明 `CARD_TEMPLATE` 是展示模板，不是玩法模板。
- `EntitlementPool` 相关 adapter：说明权益池是资源引用，不是玩法定义。

不建议写的注释：

- 在每个页面反复解释六类模板。
- 在 `rules` JSON 旁边堆业务政策说明。
- 用 TODO 记录“以后接玩法策略”。
