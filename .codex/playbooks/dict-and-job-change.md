# Dict And Job Change

## 适用场景

以下改动必须使用本手册：

- 新增、修改、废弃 `dictType`
- 新增、修改、废弃字典项
- 任何跨页面、跨 app 的用户可见状态语义治理
- 新增定时任务
- 修改任务调度参数
- 修改任务执行语义、同步语义、重试语义、并发策略或日志语义

本手册不只管"字典表和任务表"，还管"用户能看到的稳定语义来源"。

通用高风险、跨端消费细节以根 `AGENTS.md` + 对应子 `AGENTS.md` 为准。本手册只补"字典治理"与"任务治理"两个跨模块专题。命中时按 `docs/governance/AGENT_OUTPUT_PROTOCOL.md` 输出，治理链路、跨端一致性、后台任务页联动处需要 `逻辑矫正` 与 `注释审查`。

## 一、字典与用户可见语义治理

### 1. 先判断这是不是应该纳入统一语义治理

以下任一情况，默认不能散落在页面里临时写数组：

- 多个页面都会展示
- admin-web 与 miniapp 都会展示
- 会出现在搜索、筛选、列表、详情、导出、日志、统计卡中的任一组合
- 会被运营、客服、审计或任务页反复使用

如果满足以上条件，必须有统一语义来源：

- 优先字典治理
- 若不适合做成字典，也必须有共享常量、统一映射函数或统一元数据来源

### 2. 字典治理链路

涉及 `dictType` 或字典项时，至少同步更新：

1. `libs/common-constants/src/dict-governance/registry.ts`
2. `apps/backend/prisma/seed.ts`
3. 仍在使用的 init SQL 源

不允许只改其中一处。

### 3. 后端语义要求

必须确认：

- `dictType`、`dictValue`、后端枚举和数据库语义一致
- 状态 code 稳定，不把临时英文字符串当长期契约
- 若返回 `xxxText`，它必须是稳定、已本地化、可直接展示的语义，而不是临时拼装

禁止：

- 只改前端中文文案，不评估后端 code 和导出、筛选、日志、审计面
- 让原始英文枚举或数据库值直接流入用户可见界面

### 4. 前端消费要求

admin-web：

- 优先通过 `useDict`
- 或 `dict-code`
- 若不是字典，也必须有集中式 `statusMeta` / `statusMap` / `resolveXxxStatus`

miniapp：

- 必须有统一映射来源
- 页面、组件、详情卡、进度组件、结果页不得各写一套

禁止：

- 新增硬编码字典选项数组
- 同一状态在列表显示中文，在详情显示英文，在筛选里又是另一套中文
- 因为赶工直接渲染 `statusText`、`teamStatusText`，却不确认来源

### 5. 一致性检查面

每次字典或状态语义改动，至少评估以下展示面：

1. 搜索条件
2. 列表列
3. 详情页
4. 弹窗 / 抽屉
5. 导出
6. 日志 / 审计 / 统计卡
7. admin-web 与 miniapp 的跨端一致性

## 二、定时任务治理

### 1. 统一纳管

新增业务定时任务必须同时接入：

- `@CodeManagedJob`
- `@Task`

禁止只写裸 `@Cron` 就算完成。

### 2. 任务定义字段要求

必须明确：

- 全局唯一且稳定的 `key`
- 中文任务名称
- 分组
- cron 表达式
- 并发防护策略
- 可编辑策略
- 是否支持手动触发

`key` 与 `@Task.name` 必须保持一致。

### 3. 执行语义要求

必须具备：

- 幂等保护
- 并发保护
- 超时与失败处理
- 明确的单条失败策略
- 可恢复策略
- 日志与可观测性

禁止：

- 失败后既没有日志，也没有状态反馈
- 批处理任务单条失败直接吞掉，或无故中断整个批次
- 只写调度逻辑，不评估后台任务页与日志页一致性

### 4. 运维与后台同步要求

必须满足：

- 后台任务管理页面可见
- 可手动触发
- 执行状态与任务日志一致
- 调度变更不会让后台展示与实际行为分裂

## 三、验证要求

### 字典与语义治理验证

推荐执行：

```powershell
node scripts/check-dict-governance.mjs
node scripts/verify-monorepo.mjs
```

并人工确认：

- admin-web 展示面一致
- miniapp 展示面一致
- 没有原始英文 code 直出

### 定时任务验证

至少执行：

```powershell
pnpm --filter @apps/backend test -- <scheduler 或相关 service 测试>
```

若涉及后台任务管理页、日志页或同步机制，还要补 admin-web 相关验证。

## 四、停止条件

出现以下任一情况，不应宣称改动完成：

- 字典未登记治理清单
- seed 或 init SQL 未补齐
- 前端仍有硬编码字典选项或散落状态映射
- backend 仍可能把原始英文 code 暴露给用户
- 任务未纳管
- 任务不可观测
- 任务不可手动触发
- 没有对应验证建议或验证证据

## 五、常见返工点

- 只改 admin-web，不改 backend / registry / seed
- 只改某一个页面的中文文案，不治理同语义的其他页面
- 同一状态在 admin-web 与 miniapp 展示不一致
- 新任务只写了调度，没有写后台可见性与日志语义
- 把“后续再把状态收敛成字典”当作默认做法
