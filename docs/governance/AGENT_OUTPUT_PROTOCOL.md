# AI 输出协议（AGENT_OUTPUT_PROTOCOL）

## 1. 目的

本协议规定 AI 在本仓库中处理分析、方案、review、Mock、测试、实施与交付类任务时的输出规则。目标是让输出能落到真实代码、真实流程、兼容性、验证与风险闭环，而不是只复述代码或机械套模板。

文件分工：

| 文件                                                | 职责                               |
| --------------------------------------------------- | ---------------------------------- |
| `docs/governance/AGENT_OUTPUT_PROTOCOL.md`          | 入口协议、强制规则、分级、模板索引 |
| `docs/governance/AGENT_OUTPUT_TEMPLATES.md`         | 完整模板正文                       |
| `docs/governance/AGENT_OUTPUT_PROTOCOL_EXAMPLES.md` | 最小示例骨架                       |

## 2. 使用流程

每次任务先判断：

1. 用户要的是分析、方案、review、Mock、测试、实施，还是交付。
2. 涉及 `backend`、`admin-web`、`miniapp-client`、`libs`、`scripts`、`docs` 哪些范围。
3. 是否 cross-app。
4. 是否命中高风险域。
5. 是否包含长耗时、高吞吐、批量、文件、报表、Worker 或队列任务。
6. 是否涉及状态机、事实源、跨域副作用或补偿。
7. 是否需要先停手确认。

读取顺序：

1. 先读本文件确认分级、门禁和模板编号。
2. 需要套模板时，再跳到 `docs/governance/AGENT_OUTPUT_TEMPLATES.md` 的对应编号。
3. 需要骨架参考时，再看 `docs/governance/AGENT_OUTPUT_PROTOCOL_EXAMPLES.md`。

## 3. 输出分级

| 等级    | 适用场景                                                                                                                                  | 输出要求                                                           |
| ------- | ----------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| L1 轻量 | 低风险、单点、无跨 app、无契约、无状态机、无异步任务                                                                                      | 保留硬块，每块 1-3 行；页面类仍保留极简 Mermaid                    |
| L2 标准 | 普通页面、普通后端模块、普通函数或接口改动                                                                                                | 使用对应专项模板；写清调用链、数据流、状态流、直接上下游、测试建议 |
| L3 深度 | cross-app，支付/资金/认证/租户/Prisma/字典/任务，订单/退款/库存/佣金/钱包，队列/Worker/批量/报表/文件处理/数据修复，状态机/幂等/并发/补偿 | 组合多个模板，显式写出证据、不确定性、风险、验证缺口               |

L1 可以压缩，但不能删除 `逻辑矫正`、`注释审查与注释方案`、兼容性影响、测试与回归建议。

### 3.1 Harness 短交付（T0/T1）

命中 `.codex/playbooks/harness-workflow.md` 且 Tier 为 T0/T1 的实施交付，**不必**套模板 28 全文，须含：

| 字段 | 说明 |
| --- | --- |
| `tier` | T0 / T1 / T2 / T3 / PR |
| `commands_run` | 实际执行命令与 exit code |
| `commands_skipped_and_why` | 未跑 verify/全量 test 等原因 |
| 已完成 / 未验证 / 残余风险 | 禁止未跑命令写「已通过」 |

删除分级见 `.codex/playbooks/risk-tiering.md`（D0 用户已授权则禁止二次确认）。

## 4. 强制规则

### 4.1 统一硬要求

命中本协议的任务，输出中不得省略：

- 关键代码位置
- 真实流程
- 必要时的 Mermaid 流程图或影响链路图
- `逻辑矫正`
- `注释审查与注释方案`
- 兼容性影响
- 测试与回归建议

### 4.2 真实代码路径规则

- 已检索代码并确认：可以写“确认改动文件”。
- 未检索代码或证据不足：只能写“预计改动区域”。
- 不允许编造真实文件路径、行号、命令输出或验证结果。

### 4.3 高风险域必须先分流

命中根 `AGENTS.md` 高风险清单时，按任务模式区分：

- **写入 / 修复 / 迁移 / 契约 / 业务行为变更**：必须先按根 `AGENTS.md` §3.2 停手确认，再进入实施。
- **review-only 只读分析（审计、风险扫描、问题定位但不修改）**：**不需要**停手确认即可读取与分析，但输出必须：① 标记 L3 证据等级；② 写清已读证据路径（文件 + 行号或代码段）；③ 单列“未验证风险 / 推荐验证”；④ 一旦后续转入写入或修复，回到本节按写入流程停手确认。

本协议的高风险模板只用于补充分流、需求反驳和验证面，不替代根 `AGENTS.md` 的强制确认流程。

高风险域包括但不限于：

- finance / payment / auth
- tenant / 多租户
- Prisma schema / migration / seed
- 字典治理
- 定时任务
- 队列 / Worker
- 跨 app 契约
- 订单 / 支付 / 退款 / 库存 / 佣金 / 钱包
- 批量导入导出 / 报表 / 文件处理
- 数据修复 / 回填 / 迁移
- 状态机 / 幂等 / 并发 / 补偿

### 4.4 逻辑矫正是硬要求

`逻辑矫正` 不是可选建议。至少判断：

1. 当前实现是否与业务语义一致。
2. 当前职责分层是否合理。
3. 当前参数、状态、返回值、文案、字典、注释是否漂移。
4. 是否需要同步收敛直接上下游。

禁止：

- 只描述“现在代码怎么跑”，不分析“应该怎么跑”。
- 发现流程断点、状态冲突、职责错位后不单列指出。
- 进入实施后只改表面点位，不处理直接上下游的明显问题。

### 4.5 注释要求是硬要求

`注释审查与注释方案` 必须单列。至少判断：

1. 是否存在缺失注释。
2. 是否存在误导、过时或与代码不一致的注释。
3. 是否存在逐行复述代码的低价值注释。
4. 哪些位置必须补注释。

进入实施后，以下注释调整属于交付范围，不是可选项：

- 数据流和状态切换
- 边界条件与失败语义
- 兼容逻辑与历史包袱
- 事务、锁、租户来源、权限保护
- 页面复杂联动、提交流程、映射逻辑
- 队列 / Worker 幂等与补偿策略

### 4.6 关联闭环

进入实施阶段时，默认按“关联闭环”处理：

- 修复当前改动点。
- 同步修直接上下游。
- 不扩张为无关重构。

若选择不收口直接上下游，必须明确说明原因和残余风险。

### 4.7 Mock 输出规则

- 用户要求“直接可用 Mock”或“只要 JSON”时，最终只输出 JSON，不在 JSON 外混入说明。
- 用户要求解释覆盖面时，先输出“覆盖说明”，再单独输出 JSON。
- 若 JSON 要被接口直接消费，不添加 `_meta`、`coverage` 等非契约字段。

## 6. 模板组合规则

主模板解决任务核心，辅助模板补充风险面：

- 页面为主：页面梳理；辅助接口契约、Mock、菜单权限、交付。
- 后端能力域为主：后端模块梳理；辅助函数改动、接口契约、交付。
- DTO / VO / API 契约变化：接口 / DTO / VO / 契约改动；辅助后端模块、页面、`generate-types`、交付。
- 函数签名或 service 方法变化：函数改动分析；辅助后端模块或页面影响。
- 方案未定：方案设计；辅助需求澄清、后端、页面、接口。
- 字典 / 状态语义治理：接口改动或方案设计；辅助事实源、页面展示一致性、交付。
- 菜单 / 权限 / 后台入口：菜单-权限-接口一致性；辅助页面梳理、后端权限、契约改动。
- 定时任务：后端模块梳理；辅助长耗时任务隔离、任务治理、后台任务页、验证门禁。
- 队列 / Worker / 报表 / 导入导出 / 文件处理：长耗时 / 高吞吐任务隔离；辅助性能优化、SQL、页面、Worker、交付。
- 订单 / 支付 / 库存 / 退款 / 佣金 / 钱包：高风险交易链路 / 可靠性审查；辅助后端模块、接口、页面、Harness、交付。
- 数据修复 / 回填 / 迁移：数据修复 / 回填 / 迁移治理；辅助高风险门禁、交易状态机、SQL、Harness、交付。
- 工程治理：工程治理 / AI 开发链路变更审查；辅助文档蓝图、验证门禁、交付。
- 文档沉淀：文档蓝图 / 知识沉淀；辅助工程治理、交付。
- 实施完成：交付说明；辅助引用本次命中的分析模板结论。
- 会话结束：大任务用 `HANDOFF.md` + `session-orchestration.md`；决策备忘才用模板 29。
- 产品交付：`docs/delivery/README.md`（Feature Pack）+ `exec_plan` 字段关联 active plan。

### 5.1 会话编排（大任务强制）

命中 `session-orchestration.md` 中「必须建 exec-plan」或「必须新开对话」时：

- 交付（模板 28）**必须**含 `## 开发者下一步`、`## 下一会话 Prompt`（见 `AGENT_OUTPUT_TEMPLATES.md` §7.1 §11–§12）。
- 会话结束（模板 29）与 `HANDOFF.md` 二选一主格式，字段不与 HANDOFF 矛盾。
- `TaskComplete` 判定见 `HARNESS_ROADMAP.md` §0.2；M3 前不得仅凭 `check:slice` 宣称 Phase 完成。

## 7. 使用约束补充

- Review 模式必须 findings 优先；没有发现问题时明确说没有 findings，并说明剩余测试缺口或残余风险。
- 文档类任务不得绕过 `docs/governance/DOCUMENT_POLICY.md`；新增 `.md` 前必须先获得用户确认。
- 跨 app 契约变更不得跳过 `pnpm generate-types`，也不得在前端手写与 backend 同义的 DTO / VO。
- 高风险任务的“必须验证”不是完成声明；只有实际执行并通过后，交付说明才能写“已通过”。
