# 2. 总体取舍

推荐路线仍然是：

```text
C-compatible B
```

含义：

- 用完整领域模型约束方向。
- 用阶段化实施深度落地。
- 不复制一套重型活动中台。
- 不让拼课复杂度拖重所有玩法。
- 不在没有解释、事件、账务边界前继续堆营销模板。

当前项目仍处于本地开发阶段，适合先修地基。但“本地开发”不代表可以不管账务边界，恰恰相反，积分 lot、事件目录、拼课状态轴这类基础越晚上线后越难补。

# 3. 推荐实施顺序

推荐顺序：

```text
S0 文档冻结与评审
S1 P1 场景出数解释协议架构化实施
S2 P2 事件目录代码白名单
S3 P4 营销模板分类与边界冻结
S4 P5 拼课状态与事件对齐
S5 P3 积分资产账目标闭环决策
S6 后台诊断与对比能力
S7 高风险 schema 改造与迁移
```

先做 S1 / S2 / S3 / S4 的原因：

- 不必先动 Prisma。
- 能立刻提升可解释和排障能力。
- 能阻止后续玩法继续漂移。
- 能为 S5 的积分账务改造提供事件、模板边界和解释基础。

S5 之后是否进入 S7，要看项目是否确认：

```text
积分要作为长期资产账，而不只是营销余额。
```

# 4. S0 文档冻结与评审

目标：

- 冻结营销模块的目标语言和边界。
- 确认后续不再讨论“纯 B”或“一步到完整 C”。

改哪些模块：

- 仅文档。
- 可补充术语表或 ADR。

不改哪些模块：

- 不改 backend。
- 不改 admin-web。
- 不改 miniapp-client。
- 不改 Prisma。

数据表影响：

- 无。

前后台影响：

- 无运行时影响。

风险：

- 团队未对“场景出数、事件目录、积分资产账、拼课样板”形成一致理解，后续实现会再次漂移。

验证方式：

- 用现有优惠券、积分、营销模板、拼课逐一映射到 14 号文档六层模型。
- 确认每个后续需求都能归入 P1 / P2 / P3 / P4 / P5 中的某一类。

是否需要迁移历史数据：

- 否。

# 5. S1 场景出数解释协议架构化实施

对应文档：

- [15-P1-营销场景出数解释协议](./15-P1-营销场景出数解释协议.md)

目标：

- 先解决“不同手机 / 不同账户 / 不同渠道首页商品差异为什么发生”。
- 让场景出数具备 `traceId / releaseNo / source / reason` 的可持续解释能力。

改哪些模块：

- `apps/backend/src/module/client/marketing/scene`
- `apps/backend/src/module/marketing/resolution`
- `apps/miniapp-client/src/api/marketing.ts`
- `apps/miniapp-client/src/composables/use-scene-marketing.ts`
- 后续可扩展 admin 诊断页面。

不改哪些模块：

- 不改推荐算法。
- 不改订单支付退款。
- 不改优惠券账务。
- 不改积分账务。
- 不改拼课状态机。

数据表影响：

- 第一阶段不新增表。
- 先复用日志、Redis、现有审计 JSON 或内存 trace collector 结构。
- `mkt_scene_resolve_trace` 只作为后续高风险 schema 方案。

前后台影响：

- C 端响应可增加 `traceId / source`，需要走契约生成和前端适配。
- 小程序缓存 key 要纳入会员、场景、渠道、发布号或等价版本。
- 后台暂时可先无页面，先有后端 trace 查询能力。

风险：

- trace 记录拖慢首页。
- trace 包含敏感规则。
- 前端缓存维度不全导致同账户不同手机差异不可解释。

验证方式：

- 同一 `memberId + sceneCode + channel + releaseNo` 短时间内结果稳定。
- 不同账户差异能归因到受众、权益、库存、容量或缓存。
- fallback / stale-cache 能被识别。
- 模块超时能记录为降级原因。

是否需要迁移历史数据：

- 否。

# 6. S2 事件目录代码白名单

对应文档：

- [16-P2-营销事件目录与规则触发边界](./16-P2-营销事件目录与规则触发边界.md)

目标：

- 把 `MarketingEventType` 从普通 enum 升级为事件目录。
- 明确哪些事件能触发规则，哪些只是技术事件。

改哪些模块：

- `apps/backend/src/module/marketing/events`
- `apps/backend/src/module/marketing/resolution/sample-event.catalog.ts`
- 触点编排对事件目录的引用。
- 后续积分任务、发券任务、拼课事件可逐步接入。

不改哪些模块：

- 不把订单支付 / 退款强一致链路改成异步事件。
- 不允许运营新增事件。
- 不做数据库驱动事件规则。
- 不做全量事件溯源。

数据表影响：

- 第一阶段不新增表。
- 先新增 TypeScript 常量 `MarketingEventCatalog`。
- `mkt_event_catalog / mkt_event_rule / mkt_event_command_log` 后续再评估。

前后台影响：

- 后台可先无页面。
- 若做只读页面，只展示事件目录，不开放运营新增事件。

风险：

- 事件粒度过粗，规则不可表达。
- 事件粒度过细，规则爆炸。
- 资产流水事件触发同类资产变更，造成循环奖励。

验证方式：

- 每个 `MarketingEventType` 都有 catalog 定义。
- 配置事件不能触发用户权益。
- 模拟事件不能触发真实发券 / 发积分。
- 订单支付和退款仍保留直调资产服务。

是否需要迁移历史数据：

- 否。

# 7. S3 营销模板分类与边界冻结

对应文档：

- [20-P4-营销模板分类与边界方案](./20-P4-营销模板分类与边界方案.md)

目标：

- 明确项目里的“营销模板”不是单一玩法模板。
- 区分玩法能力、玩法配置模板、门店玩法配置、权益模板、展示模板、场景编排模板。
- 让后台“新增模板”不再被误解成“新增可执行玩法”。

改哪些模块：

- 先改文档和后续 TypeScript catalog。
- 后续可能涉及：
  - `apps/backend/src/module/marketing/play/play.registry.ts`
  - `apps/backend/src/module/marketing/template/**`
  - `apps/backend/src/module/marketing/config/**`
  - `apps/admin-web/src/views/marketing/template/**`
  - `apps/admin-web/src/views/marketing/config/**`

不改哪些模块：

- 不直接改 Prisma。
- 不把优惠券、积分、场景策略合进 `PlayTemplate`。
- 不开放运营新增技术玩法。
- 不把 `CARD_TEMPLATE` 和玩法模板混用。

数据表影响：

- 第一阶段无。
- 后续如补 `templateCategory / executionMode / registryCode`，命中 Prisma schema 高风险。

前后台影响：

- 后台要区分可执行玩法模板和普通表单模板。
- 门店玩法配置页只应展示可执行玩法。
- 场景模块继续引用 `MktPolicy(policyType = CARD_TEMPLATE)`。

风险：

- 继续把 `PlayTemplate` 当成完整玩法，会导致 `PT_xxx` 模板无法执行。
- 过早拆表会带来迁移成本。
- 展示模板、权益模板、玩法模板混用会污染 P1 出数解释。

验证方式：

- `COURSE_GROUP_BUY / FLASH_SALE / MEMBER_UPGRADE` 能识别为可执行玩法。
- `PT_xxx` 不能被误选为可执行玩法。
- 优惠券模板、积分规则不进入门店玩法模板选择器。
- `cardTemplateCode` 只引用 `CARD_TEMPLATE` 策略。

是否需要迁移历史数据：

- 否；若后续加字段再单独评估。

# 8. S4 拼课状态与事件对齐

对应文档：

- [18-P5-拼课复杂玩法样板化方案](./18-P5-拼课复杂玩法样板化方案.md)

目标：

- 固定拼课四条轴线：交易、成团、履约、资源。
- 把拼课事件补入事件目录。
- 把拼课不可加入、补位、成团、排课、考勤原因纳入场景解释。

改哪些模块：

- `apps/backend/src/module/marketing/course-group`
- `apps/backend/src/module/marketing/play/course-group-buy.service.ts`
- `apps/backend/src/module/marketing/events`
- `apps/backend/src/module/marketing/resolution`

不改哪些模块：

- 不拆当前拼课表结构。
- 不把虚拟成员变成订单成员。
- 不让虚拟成员进入考勤、佣金、积分奖励。
- 不让拼课履约模型污染秒杀、会员升级、签到、领券。

数据表影响：

- 第一阶段不新增表。
- 继续使用 `PlayInstance.instanceData.courseGroupTeam` 的 append-only audit。
- 继续使用 `CourseGroupBuyExtension / CourseSchedule / CourseAttendance`。

前后台影响：

- C 端展示要能区分可开团、可参团、不可加入原因。
- 后台 / 门店端继续强化团队详情、虚拟补位、排课、考勤、关闭团队。

风险：

- 虚拟补位合规风险。
- 成团和履约混淆。
- 退款回滚破坏已发生履约事实。

验证方式：

- 虚拟补位导致成团时 `formedByVirtual = true`。
- 真实人数和有效人数口径不同。
- 虚拟成员不能考勤。
- 已结课后退款不回滚履约事实。
- 成团、排课、上课、结课能被解释。

是否需要迁移历史数据：

- 否。

# 9. S5 积分资产账目标闭环决策

对应文档：

- [17-P3-积分资产账与退款原路回退方案](./17-P3-积分资产账与退款原路回退方案.md)

目标：

- 决定积分系统继续作为基础余额系统，还是升级为资产账。
- 如果升级，先明确 lot / allocation 的目标闭环和阶段切换条件。

改哪些模块：

- 先只做设计评审。
- 后续可能涉及：
  - `apps/backend/src/module/marketing/points/account`
  - `apps/backend/src/module/marketing/points/scheduler`
  - `apps/backend/src/module/marketing/integration`
  - admin-web 积分账户和流水页面。

不改哪些模块：

- S11 已确认并落地积分 lot / allocation schema；后续继续追加 Prisma 改造仍需单独确认。
- S11 已确认并接入订单积分抵扣退款链路；不在同一轮扩展到资金、支付或券退款重构。
- S11 只把历史当前可用/冻结积分初始化为 lot，不反推历史已消费积分分摊。

数据表影响：

- 如果只选基础余额系统：无。
- 如果选资产账：新增 `MktPointsLot / MktPointsFreezeAllocation / MktPointsConsumeAllocation`，可选 `MktPointsRefundAllocation / MktPointsDebt`。

前后台影响：

- 后台需要展示 lot、消费分摊、退款去向。
- C 端可先不展示 lot 明细，只展示余额和过期提醒。

风险：

- Prisma schema 改动是高风险。
- 历史数据无法精确反推原消费分摊。
- 部分退款、已过期来源退款、消费赠送积分扣回都有政策风险。

验证方式：

- 多 lot 消费 FIFO。
- 下单冻结绑定 lot。
- 支付消费沿用冻结分摊。
- 取消释放原 lot。
- 全额退款回原 lot。
- 部分退款按比例回原 lot。
- 消费赠送积分不足扣回时形成欠账或风险记录。

是否需要迁移历史数据：

- 本地开发数据可考虑重置，但仍需确认。
- 生产或准生产数据必须单独迁移方案。

# 10. S6 后台诊断与对比能力

目标：

- 让客服、测试、运营能回答“为什么不同”和“哪里失败”。

改哪些模块：

- backend admin 查询接口。
- admin-web 诊断页面。
- 可能包括：
  - 场景 trace 查询。
  - 两个 trace 对比。
  - 事件目录只读页。
  - 拼课团队解释页。
  - 积分订单抵扣来源页。

不改哪些模块：

- 不开放运营新增事件。
- 不开放运营手改积分 allocation。
- 不开放运营直接篡改拼课 audit。

数据表影响：

- 取决于 S1 / S2 / S3 / S4 / S5 是否引入独立表。

前后台影响：

- admin-web 新增诊断入口。
- miniapp 只需提供 traceId 或 source，不暴露内部规则。

风险：

- 解释信息包含敏感策略。
- 页面过度暴露风控、补位、权益细节。

验证方式：

- 用 traceId 查场景出数。
- 对比同账户不同手机差异。
- 查拼课团成团原因和虚拟补位 audit。
- 查订单积分抵扣来源和退款回退去向。

是否需要迁移历史数据：

- 否；历史旧数据可显示“旧账无结构化解释”。

# 11. S7 高风险 schema 改造与迁移

目标：

- 在确认必要性后，再进入表结构、迁移、回填和完整验证。

可能涉及：

- `mkt_scene_resolve_trace`
- `mkt_event_catalog`
- `mkt_event_rule`
- `mkt_event_command_log`
- `mkt_points_lot`
- `mkt_points_freeze_allocation`
- `mkt_points_consume_allocation`
- `mkt_points_refund_allocation`
- `mkt_points_debt`
- 拼课专用团队事实表。

不改哪些模块：

- 不一次性把所有表都加上。
- 不修改已有 migration。
- 不在没有迁移方案时处理历史数据。

数据表影响：

- 命中 Prisma schema / migration 高风险，必须按仓库规则单独确认。

前后台影响：

- backend DTO / VO 如有契约变化，必须 `pnpm generate-types` 后再适配前端。
- admin-web / miniapp-client 不得手写重复类型绕过契约。

风险：

- 迁移失败。
- 旧数据解释不准确。
- 新旧订单退款策略不同。
- 全链路验证成本上升。

验证方式：

- 新 migration。
- backend typecheck。
- 契约生成。
- 受影响前端 typecheck / lint。
- 针对迁移脚本做 dry-run 和对账。

是否需要迁移历史数据：

- 视具体表决定。
- 积分 lot / allocation 基本需要迁移或重置。
- 场景 trace、事件目录可从新增后开始记录，不迁移历史。

# 12. 不做事项

当前阶段明确不做：

- 不做完整活动中台。
- 不做运营动态新增事件。
- 不做全量规则引擎。
- 不把所有奖励统一抽成一个大 Reward。
- 不把拼课排课、考勤、虚拟补位塞进通用玩法模板。
- 不让前端二次过滤营销商品后又要求后端解释。
- 不把首页差异直接归因成推荐算法。
- 不让积分获得事件触发再次送积分。
- 不用异步事件替代订单支付、退款、券核销、积分抵扣。
- 不在未确认前继续追加 Prisma schema / migration / seed；S11 已确认的积分 lot / allocation 迁移除外。

# 13. 决策门槛

## 12.1 进入 Prisma 改造前

必须先确认：

- 是否有历史数据需要保留。
- 是否允许本地测试数据重置。
- 是否有生产或准生产数据。
- 是否要支持严格积分原路回退。
- 是否接受旧订单显示“无来源分摊”。
- 是否要新增后台诊断页面。

## 12.2 进入前端改造前

必须先确认：

- backend DTO / VO 是否变化。
- 是否需要 `pnpm generate-types`。
- admin-web 和 miniapp-client 是否都受影响。
- 小程序缓存 key 是否变更。
- 是否存在用户本地旧缓存兼容问题。

## 12.3 进入拼课退款改造前

必须先确认：

- 未成团退款策略。
- 已成团未排课退款策略。
- 已排课未上课退款策略。
- 已上课退款策略。
- 已结课售后策略。
- 虚拟补位是否随退款撤销。
- 消费赠送积分不足扣回如何处理。

# 14. 推荐首批实施包

如果只选一个首批可执行包，推荐：

```text
S1.1 场景 traceId / source / cache key
S2.1 MarketingEventCatalog 代码白名单
S3.1 模板分类 catalog 和可执行模板筛选
S4.1 拼课四轴状态说明和事件对齐
S5.1 积分资产账决策评审
```

这个包的特点：

- 基本不动数据库。
- 能显著降低后续设计漂移。
- 能为后续高风险改造提供证据。

不建议最先做：

- 直接新增积分 lot 表。
- 直接做后台大页面。
- 直接重构 PlayTemplate。

# 16. 逻辑矫正

需要先纠正几个容易误判的点：

1. “现在是本地开发，所以可以先随便做”：不准确。本地阶段最适合修地基，尤其是事件、解释、积分账务边界。
2. “先做页面，后补模型”：不建议。营销页面会放大模型问题，尤其是场景差异和拼课退款。
3. “直接上 C 方案最彻底”：不准确。完整 C 会拖重领券、签到、秒杀等简单玩法。
4. “只做 B 最快”：短期快，后续积分迁移、场景解释、拼课履约会更贵。
5. “P1/P2/P3/P4/P5 可以并行乱做”：不建议。P1/P2/P4/P5 先收边界，P3 schema 改造应后置确认。
